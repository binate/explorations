## Theme

Every defect here is a wrapper-transparency or path-parity violation in the LLVM *emitter* layer: a Kind-dispatch that peels one wrapper but not its sibling (`TYP_NAMED` vs `TYP_READONLY`), a type-discovery walk that visits one container of types (`m.Funcs`) but not its parallel sibling (`m.Globals`), and an indirect-call result lowering that already carries the byval/ptr arg convention but waits on the front end to hand it a well-formed multi-return tuple type. Two of the four cited defects turn out to be already-fixed-on-LLVM or front-end-blocked (so Plan 2's job there is verify-after-Plan-1, not re-fix); the two genuinely-live codegen gaps (Defects 1 and 2) are small, disjoint, and each fills an empty test-matrix cell.

## Defects

### Defect 1: Package-level global of a NAMED-over-aggregate/iface/func/managed-ptr type emits an invalid zero token

**Symptom.** A global whose type is `TYP_NAMED` over a non-integer underlying (`type Celsius float64; var C Celsius`, `type MyErr @errors.Error; var X MyErr`, named-over-struct/array/func/managed-ptr) emits the right *printed* LLVM type (because `llvmType` peels the name) but the wrong *init token*: `double 0` / `[3 x i64] 0` / `%BnIfaceValue 0` / `i8* 0` instead of `0.0` / `zeroinitializer` / `null`. clang rejects the non-integer cases (`integer constant must have integer type`).

**Root cause (confirmed).** `emit.bn:173-176` — the static-zero dispatch peels only `TYP_READONLY`:

```
var gt @types.Type = g.Typ
for gt != nil && gt.Kind == types.TYP_READONLY {
    gt = gt.Elem
}
```

then branches on `gt.Kind` (`emit.bn:177-196`). It never peels `TYP_NAMED`. Meanwhile `llvmType` (`emit_types.bn:53`) peels BOTH `TYP_READONLY` (`:59`) and `TYP_NAMED`→`.Underlying` (`:61-62`), so the type printed at `emit.bn:157` (`llvmType(g.Typ)`) is the underlying's, but the token chosen from `gt.Kind` is the wrapper's (`TYP_NAMED`), which matches no branch and falls through to the bare ` 0` at `emit.bn:195`. This is the exact READONLY-token bug that was already fixed here for the readonly wrapper (todo "Global var of an interface-value / func-value (or readonly-wrapped aggregate) type", binate `5dddef7d`, present in this tree at `emit.bn:167-191`) — the NAMED sibling of the same wrapper-transparency defect was left.

**Dependency / reachability caveat (must flag, do not silently route around).** The defect-of-record entry (claude-todo.md, "Package-level global of a NAMED type miscompiles", confirmed latent) locates the *reachability* root one layer up: IR-gen `resolveTypeExpr` (`gen_util.bn:294`) currently falls back to `types.TypInt()` for a `TEXPR_NAMED` that isn't a primitive/module-struct/registered-alias, so today the global's recorded `Typ` for a named-over-aggregate is plain int and codegen never even sees a `TYP_NAMED` here. That `resolveTypeExpr` fix is **Plan 1's** (front-end type resolution), not Plan 2's. So this codegen NAMED-peel is a real wrapper-transparency gap that becomes *load-bearing* only once Plan 1 makes named globals carry their true type. The two are complementary and disjoint by file; land the codegen peel regardless (it is correct and pins the invariant), and note the cross-plan ordering so the end-to-end conformance cell isn't expected green until both land. The named-scalar `i64`-by-luck case (`type Byte8 int8` → `i64`) is also a width bug fixed by the same peel.

**Fix shape.** Peel `TYP_NAMED` (recursively) alongside the `TYP_READONLY` loop before the branch selection, so the token is chosen from the same kind set `llvmType` uses. Cleanest: replace the inline READONLY-only loop at `emit.bn:173-176` with a single call to a helper that peels both wrappers — reuse the existing `unwrapNamed` (`emit_types.bn:45-50`) composed with a readonly-peel, or add one small `stripWrappers(t)` helper in `emit_types.bn` that loops `TYP_READONLY`→`.Elem` and `TYP_NAMED`→`.Underlying` to a fixpoint (mirroring how `llvmType`/`typeBits`/`isUnsigned`/`isFloatScalarParam` all peel both). Route the init-token choice through that one helper so a future wrapper can't reopen the gap at this site while `llvmType` keeps peeling it. Behavior must be byte-identical to today for the unwrapped cases (null / zeroinitializer / 0.0 / 0); only the wrapped inputs change.

**Files/functions.** `pkg/binate/codegen/emit.bn` (the global-var static-zero dispatch, `EmitModule`, the loop+branch at ~:173-196 — the only edit). Optionally `pkg/binate/codegen/emit_types.bn` (add/extend a `stripWrappers`/`unwrapNamed` helper to share with `llvmType`). No other file.

**Test coverage.** GAP — the matrix cell is empty. `emit_global_test.bn` currently covers func-value, iface-value, readonly-aggregate, and float zero-init (`TestEmitGlobalFuncValueZeroInit`/`...IfaceValueZeroInit`/`...ReadonlyAggregateZeroInit`/`...FloatZeroInit`) but has **no NAMED-over-aggregate / named-scalar case**. Add codegen unit tests: `type Celsius float64; var g Celsius` → ` double 0.0` (not ` 0` / ` i64 0`); `type MyErr @errors.Error; var g MyErr` → `%BnIfaceValue zeroinitializer`; a `type Bytes3 [3]int8; var g Bytes3` → `zeroinitializer`. These are mode-independent and guard the codegen fix below the conformance layer (the same pattern the readonly sibling test uses). The end-to-end conformance readback (`type Temp float64; var g Temp = 3.5`) depends on Plan 1's `resolveTypeExpr` fix to be reachable — author it but keep it xfailed/pending until both land.

### Defect 2: Cross-module by-value struct/array global emits `external global %Struct` / `global %Struct zeroinitializer` without ever declaring `%Struct = type {...}`

**Symptom.** A module that references a struct-typed package-level global (its own definition, or an imported `var g StructType` reached as `external global`) emits `@<mangled> = [external] global %bn_<pkg>__Struct …` but never emits the matching `%bn_<pkg>__Struct = type <{ … }>` → clang `use of undefined type named '...'`. Same for `[N]Struct` arrays and named-over-struct globals.

**Root cause (confirmed).** Two adjacent gaps in `emit_types.bn`:

1. `collectStructTypes` (`emit_types.bn:128-160`) iterates ONLY `m.Funcs` (params at `:135`, results at `:141`, every instruction/arg type at `:149-154`). It never iterates `m.Globals`. Struct-type defs are accumulated into `moduleStructDefs` and emitted (`emit.bn:95-124`) *before* globals are emitted (`emit.bn:142+`), so a struct reachable only through a global var is never discovered → never declared. This is the same path-parity miss the defect-of-record entry ("Cross-module global of struct type emits `external global %Struct`…", claude-todo.md) predicts: "moduleStructDefs doesn't include types reachable only through an imported global."
2. `discoverStructFromType` (`emit_types.bn:162-183`) has recursion arms for `TYP_READONLY` (`:167`), `TYP_STRUCT` (`:171`), `TYP_MANAGED_PTR`/`TYP_POINTER` (`:175`), `TYP_SLICE`/`TYP_MANAGED_SLICE` (`:179`) — but **no `TYP_NAMED` arm** (so a named-over-struct global's struct is missed even via a function) and **no `TYP_ARRAY` arm** (so `[N]Struct` never recurses to its element). Wrapper-transparency + container-recursion both incomplete.

Confirmed today only by the unit-test harness module (no conformance test references a cross-package `var g StructType` — `548_cross_pkg_extern_var` exercises only a scalar int and a raw-slice/string global, `548/main.bn`), but the gap is real for any genuine cross-package struct-global import.

**Fix shape.** (a) In `collectStructTypes`, after the `m.Funcs` loop, scan `m.Globals`: `for i … len(m.Globals) { discoverStructFromType(m.Globals[i].Typ) }` (this covers both the defining-package `global` and the consuming-package `external global` paths, since both record `g.Typ`). (b) In `discoverStructFromType`, add a `TYP_NAMED` arm that recurses into `.Underlying`, and a `TYP_ARRAY` arm that recurses into `.Elem` — mirroring the existing READONLY/pointer/slice arms (register-then-recurse is not needed here since these wrappers carry no own struct def; just recurse). Keep the existing infinite-recursion guard (`addStructDef` registers before recursing, `:196-201`).

**Files/functions.** `pkg/binate/codegen/emit_types.bn` (`collectStructTypes` ~:128, `discoverStructFromType` ~:162). No other file (`emit.bn`'s emit loop already iterates `moduleStructDefs`, so once discovery populates it the type-def is printed before the globals).

**Test coverage.** GAP. Add (a) a codegen unit test asserting that a module with a struct-typed global emits `%bn_main__Pt = type` before the `@…__g = global %bn_main__Pt …` line (and an `external global` variant), and a `[2]Pt`-typed global variant for the `TYP_ARRAY` arm; (b) a conformance test: a 2-package program where pkg A defines `var g StructType` and pkg B imports + reads `g.field`, xfailed until the fix (mirrors `548` but with a struct type). The unit test pins the type-decl emission directly; the conformance test pins the end-to-end clang-compiles-and-runs.

### Defect 3: >16-byte byval struct arg + trailing-scalar drop through an INDIRECT (iface) call, and multi-return result through iface dispatch

**Status: the byval-arg / trailing-scalar halves are ALREADY FIXED in this tree; the live remainder is a Plan-1 (IR-gen) fix, not codegen.**

**Symptom (historical, per the probe in the prompt).** (a) A >16-byte struct passed by value through `iface.M(...)` SIGSEGV'd (value-as-pointer). (b) A multi-word byval arg followed by a scalar dropped the trailing scalar through vtable dispatch (598). (c) A multi-return iface method (`M() (int,int)` / `(T,@Error)`) emits `extractvalue void %vN, 0` → clang `void type only allowed for function results`.

**Root cause (confirmed — and what's already done).** `emit_iface_call.bn` `emitCallIfaceMethod` already carries the byval/sret aggregate convention: it emits `writeParamTypeLLVM(out, argInstr.Typ)` for the bitcast signature (`:100`), `writeByvalArgPreamble(out, instr)` to spill each >16-byte by-value aggregate into its hoisted `.bv<i>` slot (`:108`, with the explicit comment at `:103-107` naming 598 and the >16-byte SIGSEGV), and passes `ptr %v<id>.bv<i+1>` for `isByvalParam` args at the call (`:145-152`). The sret-result path is correct (`:42` `useSret = needsSret(instr.Typ)`, alloca hoisted by `emitIfaceMethodSretAllocDecl` `:181-193`, call-through-and-load `:110-169`). This matches Plan 3's "LANDED (binate `3892e7d1`)" for §3.1/§3.7. So the >16-byte byval-arg + trailing-scalar cases are **not live** here.

The remaining live failure is the **multi-return result type** (claude-todo.md CRITICAL, "Interface-method dispatch drops the result type for any method with ≠1 results"): `gen_iface_registry.bn:180` populates only the singular `MethodResults @[]@types.Type` (one type per method, `nil` for 0 or ≥2 results); `findInterfaceMethod` (`gen_iface.bn:128-180`) returns that single `rt` (nil for multi-return); `genInterfaceMethodCall` (`gen_iface.bn:63`) then builds the dispatch instr with a nil/void result type, so `instr.Typ` reaching `emitCallIfaceMethod` is void and the multi-assign's `extractvalue` is on `void`. **This entire result-type construction is in `pkg/binate/ir` — it is Plan 1's defect** (Plan 1 adds `MethodResultsFlat`/`MethodResultCounts`, generalizes `findInterfaceMethod`, and has `genInterfaceMethodCall` build the multi-return tuple type via `makeMultiReturnStructType`).

**Fix shape (Plan 2's part: verify-only, expected zero codegen change).** Once Plan 1 makes `instr.Typ` for a multi-return iface dispatch the proper multi-return tuple struct type, `emitCallIfaceMethod` already lowers it correctly: `useSret = needsSret(instr.Typ)` selects sret for >16-byte tuples (the existing alloca-hoist + load path), and small tuples (≤16B, register-returned) flow through the `else` arm at `:132-141` (`call {…} %fn(...)` + the caller's `extractvalue`). Plan 2's job is to **confirm** this on the LLVM modes after Plan 1 lands — specifically the small-tuple register-return arm (`iface-multi-return/int/2`, a 16-byte `{i64,i64}`) and a >16-byte tuple (`/int/3..5`, sret). Do not pre-emptively edit it — the byval arg side is correct and the result side is gated on Plan 1.

**Files/functions.** Verification target: `pkg/binate/codegen/emit_iface_call.bn` (`emitCallIfaceMethod`, `emitIfaceMethodSretAllocDecl`). Reuses `needsSret` (`emit_types.bn:29`), `writeParamTypeLLVM`/`isByvalParam`/`writeByvalArgPreamble` (`emit_util.bn:316/:290/:367`). No codegen edit expected.

Plan 2's two *own* commits (Defects 1 and 2) have no dependency on the seam at all and can land independently.

### Defect R2-1: `isByvalParam` doesn't peel the `readonly`/named/alias wrapper → a `readonly`-wrapped >16-byte aggregate param is lowered by-value in BOTH the callee signature and the call site (should be byval-by-pointer)

The recon/plan missed the IR copy; fixing only codegen turned the silent runtime-garbage into a hard clang error (`'%v0' ... type 'ptr' but expected '%bn_..._Big'` — signature says `ptr`, callee copy says struct value). Per the user's decision, **unified both into one `Type.IsByvalParam()` in `pkg/types`** (`scope.bn`, beside `SizeOf`), routing IR-gen + codegen (11 call sites) through it, so the "two predicates must agree" hazard can't recur. The peel resolves alias/readonly/named (matching `SizeOf`); symbol output byte-identical for unwrapped params. Tests: `types_query_test.bn TestIsByvalParamPeelsWrappers` + conformance `matrix/readonly/pass-arg/value-struct-large` (green on all LLVM modes; xfailed on VM (shared-IR readonly field-read, plan-cr2-1 Def 1) and native (the `common.IsAggregateTyp` readonly-byval gap — Plan 3, filed in claude-todo)).

**Symptom.** A parameter typed `readonly Big` (24-byte struct), `readonly [4]int`, or `readonly @[]int` is lowered as a plain >16-byte LLVM struct *value* in the callee `define` line and passed as a plain struct value at every call site — never routed through the `ptr`/byval spill convention the >16-byte ABI requires. Probe (from the todo): `func first(b readonly Big) int { return b.a }; first(x)` with `x.a=123` → garbage `6102984704` (expected 123); SIGSEGV on other shapes. Plain `Big` / `@[]int` / below-16B-readonly / alias-typed-slice controls all work — only readonly-wrapped >16-byte aggregates diverge.

**Root cause (confirmed).** `isByvalParam` (`pkg/binate/codegen/emit_util.bn:290-306`) tests `t.Kind` against the aggregate set (`TYP_STRUCT/ARRAY/SLICE/MANAGED_SLICE/FUNC_VALUE/MANAGED_FUNC_VALUE/INTERFACE_VALUE/INTERFACE_VALUE_MANAGED`) **before peeling any wrapper** — there is no `TYP_READONLY` arm and no peel. A `readonly Big` arrives as `TYP_READONLY` (Kind not in the set) → `isByvalParam` returns `false` at line 303 and never reaches the `t.SizeOf() > 16` gate at line 305. Meanwhile both `SizeOf` (`pkg/binate/types/scope.bn:114`, peels `TYP_READONLY`→`.Elem`) and `llvmType` (`emit_util.bn`→`emit_types.bn:81`, peels `TYP_READONLY`→`.Elem`) DO peel, so the printed parameter/argument LLVM type is the *underlying* aggregate (`%bn_..._Big`, by value) while the byval decision says "not byval". The two disagree: the >16-byte aggregate is emitted as a plain LLVM IR struct value (IR-level "spread") instead of the textbook indirect convention `byval` exists to force.

Every byval decision site keys on this one predicate, so the gap propagates to **both** call shapes I verified:
- direct call (`emit_call.bn`): signature via `writeParamTypeLLVM` (`emit_util.bn:316`, falls to bare `llvmType`), arg via `writeByvalArgLLVM` (`emit_util.bn:418`, falls to the else value-branch at :431-447), plus `writeByvalArgPreamble`/`emitByvalAllocDecls` (`emit_util.bn:367`, :394) skip the spill (`continue` at :370 / :397).
- indirect iface call (`emit_iface_call.bn` `emitCallIfaceMethod`): bitcast signature via `writeParamTypeLLVM` (`emit_iface_call.bn:100`); the per-arg branch at `:145` falls to the else by-value path (`:153-157`).

This is the SAME wrapper-transparency class as the already-landed pieces but in the *calling-convention* layer those fixes never touched. (The todo also names the native classifier `common.IsAggregateTyp` at `pkg/binate/native/common/common.bn:345`, which peels only `UnwrapNamed`, not readonly/alias — that native half is Plan 3's; Plan 2 owns the LLVM `isByvalParam`.)

**Fix shape.** Peel the transparent wrappers (`TYP_READONLY` + `TYP_NAMED` + `TYP_ALIAS` to a fixpoint) at the TOP of `isByvalParam`, before the `t.Kind` aggregate-set test — so the Kind test and the `SizeOf() > 16` gate see the same representation type `llvmType`/`SizeOf` already see. The peel helper already exists in this package: `stripWrappers` (`emit_types.bn:61`, landed by `f2ebaca1` for Defect 1) peels `TYP_READONLY`+`TYP_NAMED`; either extend it to also peel `TYP_ALIAS` (via `t.ResolveAlias()`, since `ResolveAlias` at `types_query.bn:120` peels only `TYP_ALIAS`) or compose `t.ResolveAlias()` then `stripWrappers`. Route `isByvalParam` through it: `t = stripWrappers(t.ResolveAlias())` (guard nil) as the first statement. Behavior must be byte-identical for unwrapped inputs (only wrapped >16-byte aggregates change shape, from value to byval). Because every byval site (`writeParamTypeLLVM`, `writeByvalArgLLVM`, `writeByvalArgPreamble`, `emitByvalAllocDecls`, `emit_iface_call.bn:145`) funnels through this one predicate, the single-site peel fixes both the direct- and indirect-call shapes at once.

**Files/functions.** `pkg/binate/codegen/emit_util.bn` (`isByvalParam` ~:290 — the only edit). Optionally `pkg/binate/codegen/emit_types.bn` (extend `stripWrappers` to also peel `TYP_ALIAS`, to share the one peel helper). No other codegen file (the call/iface sites already route through `isByvalParam`).

**Test coverage.** GAP — no existing test passes a readonly aggregate >16 B as an argument. Add (a) a codegen unit test on `isByvalParam` asserting it returns `true` for `readonly Big`(24 B) / `readonly [4]int` / `readonly @[]int` and stays `false` for `readonly` below-threshold and non-readonly controls; (b) a codegen emission test that a callee `define` for a `readonly Big` param emits the byval `ptr`-param shape and the matching call site emits the `.bv<i>` spill (mirrors the existing byval emission tests for plain >16B aggregates); (c) a conformance cell: `func first(b readonly Big) int { return b.a }` via a DIRECT call AND via an interface method, with plain/below-threshold/alias controls — xfail on LLVM until fixed. The native counterpart of (c) belongs to Plan 3.

**Relation.** Extends Plan 3's byval-arg work (`3892e7d1`, "pass >16-byte aggregate args through indirect calls by pointer") and the `isByvalParam`/`writeByvalArgPreamble` machinery it introduced — that landed the *unwrapped* >16-byte aggregate case but the readonly-wrapped sibling was missed because the Kind test precedes any peel. ### Defect R2-2: `&G`-as-rvalue STILL dropped at `OP_CAST` and at the iface-method arg site (the two LLVM value-operand sites the `emitValRef` migration missed)

**Status (2026-06-09): LANDED — binate `d086ccac`.** `emitCast` now precomputes the source via `emitValRef(buf.New(), Args[0])` (mirroring `emitBitCast`) and writes it at all 11 arms (`writeBuf` is non-destructive, so the same-size aggregate identity arm that emits the source twice stays correct); `emit_iface_call.bn:156` switched `emitRef`→`emitValRef`. Tests: conformance `669_cast_global_addr` + `670_iface_method_global_addr`, green on all LLVM modes + VM. Native halves are Plan 3 (claude-todo): `669` xfails all 3 native modes (native OP_CAST drop); `670` xfails aa64 only (native x64 iface-arg already fixed in `0c707e1f` — verified: 670 passes on x64-darwin).

**Symptom.** Two value-operand positions still render the `IsGlobalRef` pseudo (ID -1) via bare `emitRef` → `%v-1`, which has no SSA definition. (a) `var addr int = cast(int, &G)` → clang `error: use of undefined value '%v-1'` (the `ptrtoint i8* %v-1 to i64` line). (b) `i.m(&G)` (a global address passed to an interface method) → clang error on `... i8* %v-1` in the dispatched call. VM is correct (it materializes `IsGlobalRef` op-agnostically up front). These are idiomatic, type-valid programs.

**Root cause (confirmed).**
- `emit_ops.bn` `emitCast` (`:196`) fetches its source operand with `emitRef(out, instr.Args[0].ID)` at every conversion arm (`:239,:250,:266,:283,:295,:307`, …) — NOT `emitValRef`. The pointer→int arm (`:303-312`, `ptrtoint`) is the one `cast(int, &G)` takes; it emits `ptrtoint i8* %v-1 to i64`. Its sibling `emitBitCast` (`:372`) was already fixed: it precomputes `var srcRef = emitValRef(buf.New(), instr.Args[0])` (`:391`, comment at `:388-390` names `bit_cast(int, &G)`) and writes `srcRef`. `emitCast` never got that treatment.
- `emit_iface_call.bn:156` emits a non-byval iface-method scalar arg with `emitRef(out, argInstr.ID)` (the else branch of the `isByvalParam` test at `:145`) instead of `emitValRef`. The direct-call arg path (`writeByvalArgLLVM`, `emit_util.bn:446`), the indirect-call arg path (`emit_call.bn:201,:485`), the return form (`emit_helpers.bn:226,:234,:271`) and the compare operands (`emit_ops.bn:171,:173`) all already use `emitValRef`; the iface-arg site is the lone straggler.

`emitValRef` (`emit_util.bn:43`) renders the pseudo as `@<mangled>` (a valid LLVM value operand under opaque pointers, no materialization instruction needed); `emitRef` (`:13`) always emits `%v<id>`. This is the per-op whack-a-mole the todo's architectural finding calls out: the durable fix is to make `emitValRef` the SOLE value-operand fetch (audit every site) or mirror the VM's up-front materialization pass.

**Fix shape.** (a) In `emitCast`, precompute the source ref once via `emitValRef` (exactly as `emitBitCast` does) and write that buffer at every arm, replacing the per-arm `emitRef(out, instr.Args[0].ID)`. (b) At `emit_iface_call.bn:156`, replace `emitRef(out, argInstr.ID)` with `emitValRef(out, argInstr)`. While there, audit the remaining codegen value-operand sites so a future op can't reopen this (the architectural fix the todo prefers); at minimum, leave a note that `emitValRef` is the canonical value-operand fetch. Verify against the `IsGlobalRef`-true input only — the unwrapped output is byte-identical (`emitValRef` == `emitRef` for ordinary instructions).

**Files/functions.** `pkg/binate/codegen/emit_ops.bn` (`emitCast` ~:196), `pkg/binate/codegen/emit_iface_call.bn` (`emitCallIfaceMethod`, the non-byval arg branch at `:153-157`). No other codegen file. (The native `OP_CAST` source at `x64_dispatch.bn:388` / `aarch64_dispatch.bn:411` and the aa64 iface-arg are Plan 3's.)

**Test coverage.** Add WITH the fix: a conformance cell `var addr int = cast(int, &G)` round-trip (FAILS pre-fix on LLVM with `use of undefined value '%v-1'`, PASSES on VM); an `i.m(&G)` iface-method-arg cell; codegen unit pins that `emitCast`'s `ptrtoint` arm and the iface-arg site render an `IsGlobalRef` operand as `@<mangled>`, not `%v-1`. The `bit_cast(int,&G)` sibling is already covered (conformance 551); the value-preserving `cast(int,&G)` was uncovered.

**Relation.** Extends the `emitValOperand`/`emitValRef` migration whose LLVM return-form + bit_cast piece landed (`91ef4fc4`-era / `emitBitCast`'s `srcRef` precompute) and whose x64 return-form landed at `0c707e1f` ("materialize a global address used as an rvalue") — the enumeration of value-operand sites was incomplete, and `OP_CAST` + the iface-method arg are the two LLVM positions it skipped. Same `IsGlobalRef` pseudo-materialization root as the resolved x64 `return &G` (`0c707e1f`); this is the LLVM cast/iface-arg variant.

### Defect R2-3: `emitIfaceUpcast` writes `IfaceParentSlotOffset`'s −1 sentinel verbatim as a vtable GEP index — silent vtable corruption (FILED, user-DEFERRED)

**Status (2026-06-09): LANDED — binate `ca155319` (user re-authorized).** Done as the coordinated 3-plan set, not LLVM-only: (b) IR (`gen_iface_extends.bn`) `IfaceParentSlotOffset` returns 0 for the same canonical interface (removes the only emittable −1 — the zero-method `@X→*X` decay); (a) the LLVM lowering (`emit_iface_upcast.bn`) AND both native dispatch lowerings (`aarch64/x64_dispatch.bn`) hard-`panic` on a negative offset. The IR fix landed WITH the asserts (else they'd fire on the decay's −1). Asserts verified to never false-fire: the full iface suite (140 cells) green on builder-comp / -comp-comp / -comp-int / native aa64 / native x64-darwin. Tests: `gen_iface_extends_test.bn TestIfaceParentSlotOffsetSameInterfaceIsZero` + conformance `685_iface_same_interface_decay`.

**Symptom.** `OP_IFACE_UPCAST` computes the target ancestor's sub-vtable offset via `IfaceParentSlotOffset(src, target)`, which returns **−1** when `target` is not a (transitive) parent of `src` (e.g. the same-canonical-interface `(X,X)` case, which is special-cased to 0 only for `any`, not in general). The LLVM lowering feeds that −1 directly into `getelementptr inbounds i8*, i8** %vt, i64 −1` → the vtable pointer walks one slot BEFORE its base → the dispatched method slot reads the wrong word (e.g. the dtor) → call through garbage/null → SIGSEGV or silent wrong dispatch. Cross-backend divergence: LLVM writes `i64 −1` verbatim while both native backends guard with `if byteOff > 0` (`aarch64_dispatch.bn:166`, `x64_dispatch.bn:242`) and accidentally leave the pointer unchanged (offset 0) — same IR, base−1 on LLVM vs base on native.

**Root cause (confirmed).** `emit_iface_upcast.bn` `emitIfaceUpcast` (`:28`) computes `offset = ir.IfaceParentSlotOffset(...)` (`:34-37`) and writes it unconditionally as the GEP index: `.new = getelementptr inbounds i8*, i8** %v<id>.vt, i64 <offset>` (`:55-61`, `out.WriteInt(offset)` at `:60`) with **no −1 guard / assert**. The unit-test comment at `gen_iface_extends_test.bn:72-75` claims the caller "leaves offset at 0" for the −1 case, but no caller clamps it — the comment is aspirational; the code is a footgun. Currently latent: the one known emitter of a −1-offset upcast (the alias-reexport path) was fixed upstream at the IR level (`a869e8e7`, "canonicalize iface aliases before deciding upcast"), so no reachable caller hands `emitIfaceUpcast` a −1 today — but any future mis-emitted upcast becomes a memory-safety bug here instead of a loud error.

**Fix shape.** Make a −1 offset a HARD error in the LLVM lowering, never a silent GEP index: in `emitIfaceUpcast`, after computing `offset`, assert `offset >= 0` (the lowering should panic/error on −1 rather than emit `getelementptr ... i64 −1`). This is sub-part (a) of the deferred todo entry for the LLVM lowering. The complementary fixes are NOT Plan 2's: making `IfaceParentSlotOffset(X,X)` return 0 like the `any` case (sub-part b) is Plan 1 (`pkg/binate/ir/gen_iface_extends.bn`); the native `*_dispatch.bn` guards becoming asserts are Plan 3. The three lowerings should agree (all assert, none silently clamp), so this wants to land as a coordinated set across plans — another reason it stays deferred until the user decides.

**Files/functions.** `pkg/binate/codegen/emit_iface_upcast.bn` (`emitIfaceUpcast`, the offset computation + GEP at ~:34-60). No other codegen file. Plan 1 owns the `IfaceParentSlotOffset(X,X)` source fix; Plan 3 owns the native `*_dispatch.bn` guard asserts.

**Test coverage.** A codegen unit test that an `OP_IFACE_UPCAST` whose `(src,target)` yields −1 triggers the assert (rather than emitting `i64 -1`). End-to-end reachability is currently bounded (no live −1 emitter after `a869e8e7`), so a runtime conformance repro isn't presently constructible without re-introducing the alias bug — pin it at the unit level. Author and xfail/hold pending the user's deferral decision.

**Relation.** Direct follow-up of Plan-1 Defect 8 (`a869e8e7`, transitive iface re-export SIGSEGV) — that fix stopped the alias path from emitting the −1-offset upcast, but the lowering still trusts the offset blindly, so the footgun survives for any other non-parent / same-interface upcast. This is the LLVM-lowering sub-part (a) of the FILED-but-deferred `IfaceParentSlotOffset −1 sentinel` CRITICAL; the IR-level (Plan 1) and native (Plan 3) sub-parts are siblings of this same defect.

### Not a Plan-2 defect (verified): plain (no-managed-field) struct/array composite literal passed BY VALUE

Source-confirmed this is an **IR-gen (Plan 1) defect, not a codegen one** — Plan 2 has nothing to fix. `dist(Pt{a:3,b:4})` passes the alloca *pointer* instead of the value because `coerceArg` (`pkg/binate/ir/gen_call.bn:66-77`) only loads the `OP_ALLOC` aggregate inside the `if needsStructCopy(paramTyp)` branch; a plain (no-managed-field) aggregate has `needsStructCopy == false`, so the un-loaded `OP_ALLOC` flows out as the call arg. The LLVM emitter (`emit_call.bn:94` → `writeByvalArgLLVM` → `emit_util.bn:446 emitValRef`) then faithfully renders whatever IR hands it: given an `OP_ALLOC` arg it emits the pointer, given a loaded value it emits the value. There is no codegen-side change that fixes this without the IR-gen load — the todo correctly assigns **"Owner: Plan-1 (`pkg/binate/ir/gen_call.bn`)."** Recorded here only so it isn't re-triaged into Plan 2; scope confirmed (all by-value composite-literal arg call shapes — direct and indirect — bottom out at the same `coerceArg` load gate, so the fix is one IR-gen site, not a per-backend codegen change).
