# Code-Red P2 — Plan 4: Scalar &amp; float-literal value correctness (sub-word narrowing, 64-on-32, unsigned int↔float, float-const materialization, div/0 panic)

> One of four disjoint code-red P2 work plans (partition in `plan-code-red-p2.md`).
> Source-confirmed: each defect cites root cause, fix shape, files, and test status.
> Defect details are tracked in `claude-todo.md`; this plan is the execution view.

## Summary

All six defects share one root cause class: a scalar's observable value is defined by its resolved type (width + signedness), but the VM and both native backends compute in full host-word registers / always-signed FP conversions, never re-establishing the type's invariant. LLVM is correct for free (true-width SSA + isUnsigned-gated cvt). The unifying fix is plan-code-red.md §3.8's "narrow once in a shared classifier": (a) a post-op sub-word narrow keyed on result type, (b) signedness-dispatched int↔float (mirroring the cmp/div/shift selection that already exists), (c) float-const materialization at resolved width (F64BitsToF32Bits). The divide-by-zero work is the ratified-but-unimplemented OP_DIV_CHECK + unsafe_div/unsafe_rem, mirroring OP_BOUNDS_CHECK/unsafe_index exactly. Crucially, the two DEFERRED float-const blockers (539/844 float32, and the 538/777 round-bit) were filed as "blocked on a new BUILDER release" — I confirmed BUILDER is now at bnc-0.0.7, cut 2026-06-04 from a tree that already contains common.F64BitsToF32Bits (commit 8042d609, 2026-06-03, is an ancestor of the 0.0.7 bump d407a66a). So the 539 float32-const fix is very likely UNBLOCKED now and should be re-attempted, not left deferred.

## Unifying strategy

Route every coercion through a small set of type-driven classifiers, one per concern, consulted on all three non-LLVM targets identically (LLVM stays correct): (1) a "narrow sub-word result" step applied after every value-producing integer op, keyed on instr.Typ.Width/Signed — implemented per-target (VM: a BC_NARROW with width/signedness carried in BCInstr.Imm/Aux, emitted by the lowerer right after the arith op, OR a fused narrow inside execArithOp using a width field added to BCInstr; natives: a sext/zext/and-mask tail after Add/Mul/etc. gated on Width<wordbits). (2) Int↔float signedness dispatch: add the unsigned op family (VM BC_UITOF; aarch64 Ucvtf/Fcvtzu — NOT YET in the asm lib, must be added; x64 the uint64→double add-back idiom since Cvtsi2sd has no unsigned form) selected by srcTyp.Signed, exactly mirroring lowerBinOp/emitBinop's existing isUnsigned gate. (3) Float-const at resolved width: route emitConstFloat / OP_CONST_FLOAT through common.F64BitsToF32Bits when the const type is float32. (4) Divide-by-zero: one OP_DIV_CHECK emitted in genBinary + the compound-assign path before OP_DIV/OP_REM, lowering to an rt.DivCheck call on every backend, with unsafe_div/unsafe_rem builtins passing checked=false. The decisive structural point: BCInstr (vm.bni:342) and ins.Typ (natives) both already carry — or can carry — the result type, so the classifier has its input at every site; the failure was simply that no narrowing/unsigned step consulted it.

## Defects

### 4.1 Sub-word arithmetic results not narrowed in the VM and both native backends — dirty upper bits corrupt unsigned compare/shift/div/widen/return

**CRITICAL · CONFIRMED · VM LANDED 2026-06-05 (binate 435b6cdd); aa64/x64 pending**

- **Progress / decided design**: per-backend narrowing keyed on a single shared
  classifier `common.SubWordNarrow(t, wordBits) -> (width, signed)` (the user's
  call: each backend narrows in its own emitter; LLVM untouched). VM portion
  LANDED: the lowerer stamps the narrow on the arith bytecode's free Imm/Aux
  (Imm = result width, Aux = signedness) and `execArithOp` re-narrows via the
  `narrowToWidth` primitive now shared with BC_SEXT/BC_ZEXT. Dropped the
  add/sub/mul × {8,16,32} VM-lane xfails; div/rem were never xfailed (in-range
  quotient of clean operands needs no narrow); int-to-float stays (defect 4.2).
  REMAINING: aarch64 `emitBinop` sext/uxt tail (Sxtb/Sxth/Sxtw/Uxtb/Uxth +
  32-bit-mov for uxtw — encoders confirmed present) and x64 `emitBinop`
  movsx/movzx tail; then drop the native scalar-matrix xfails per lane. The
  native bit_cast(int32)/widening sign-extension residual (539 negative const)
  is the same invariant on a non-arith producer — fold it in with the native
  work so the native 539 xfail can drop too.
- **Root cause**: VM: vm_exec_pure.bn:19-43 execArithOp computes regs[Dst]=regs[Src1] op regs[Src2] at host int width with NO post-op narrowing; e.g. (uint16 a*b)>>8 leaves the un-narrowed product, so the shift reads dirty bits 16-63 (37796 on VM vs 164 on LLVM). The BCInstr (vm.bni:342) carries no width field, so execArithOp cannot narrow even if it wanted to — the lowerer (lower_instr_helpers.bn lowerBinOp:144-215) selects only the opcode, never stashing the result width. Natives: aarch64_ops.bn:53-93 emitBinop emits every op at 64-bit (aarch64.Add(a,true,...)); x64_ops.bn:90-97 emits at SZ64; both have ins.Typ available (used for the isUnsigned gate, aarch64_ops.bn:50-51 / x64 emitDivOrRem) but never narrow the result to instr.Typ.Width. LLVM is correct via true-width SSA. Latent until an op result is consumed by a width-sensitive op (shift/unsigned-cmp/div/widen/return) WITHOUT an intervening sized store/cast (which re-narrows).
- **Fix shape**: Add a single shared 'narrow to result width' step keyed on instr.Typ.Width (sub-64) and signedness. VM: extend BCInstr with the result width (or repurpose Imm/Aux on the arith ops — they currently set only Op/Src1/Src2/Dst, so Imm is free), and either (a) emit a separate BC_NARROW8/16/32 (sext for signed, zext/mask for unsigned) right after the arith opcode in lower_instr.bn's arith arm (lines 70-80), or (b) make execArithOp narrow inline using the carried width. Prefer the post-op-narrow-in-lowerer form so it composes with the existing BC_SEXT/BC_ZEXT/BC_TRUNC handlers (vm_exec_helpers.bn:167-194). Natives: after each Add/Sub/Mul/And/Or/Xor/Shl in emitBinop, when ins.Typ.IsInteger() && Width<wordbits, emit a width-narrowing tail — aarch64 SBFM/UBFM (sxtb/sxth/sxtw / uxtb/uxth + 32-bit-mov-zeroes-upper) or AND-mask; x64 movsx/movzx from the sub-width reg or AND with the width mask — gated on signedness. Decide (user call, per plan-code-red.md P3/§3.8) whether this is each backend's job or one IR-gen narrowing cast; either way it must be a single shared classifier, not per-op-arm copies. Mirror the existing isUnsigned selection pattern.
- **Files**: `pkg/binate/vm/vm_exec_pure.bn (execArithOp:19-43)`; `pkg/binate/vm/lower_instr.bn (arith arm:70-80)`; `pkg/binate/vm/lower_instr_helpers.bn (lowerBinOp:144-215)`; `pkg/binate/vm.bni (BCInstr:342 — may need a width field)`; `pkg/binate/native/aarch64/aarch64_ops.bn (emitBinop:17-95)`; `pkg/binate/native/x64/x64_ops.bn (emitBinop:39-110; emitDivOrRem)`; `pkg/binate/asm/aarch64/* and pkg/binate/asm/x64/* (sub-word narrow encoders if not present)`

### 4.2 Unsigned int→float uses a SIGNED conversion in the VM and both native backends — negative result for high-bit-set unsigned

**CRITICAL · CONFIRMED**

- **Root cause**: VM: lower_cast.bn:38-43 emits BC_SITOF for EVERY int→float cast regardless of srcTyp.Signed; vm_exec_helpers.bn:195-210 BC_SITOF does cast(float64, i) where i is host int (signed), so a uint64 with bit 63 set converts negative. Confirmed: cast(float64,<uint64 0xC000000000000000>)>0.0 is true on LLVM, false on VM. aarch64: aarch64_float.bn:207-217 emitFloatCast always emits Scvtf (signed); no Ucvtf. x64: x64_float.bn:248-258 always emits Cvtsi2sd (signed); x86 has no direct unsigned int64→double. The asm libs lack the unsigned encoders entirely (grep found Scvtf/Fcvtzs in aarch64_fp.bn:114,123 but no Ucvtf/Fcvtzu; Cvtsi2sd/Cvttsd2si in x64_fp.bn:96,102 but no unsigned form). The checker permits the cast. The float→int direction (BC_FTOSI / Fcvtzs / Cvttsd2si) is signed-only too. A uint32 is zero-extended (positive in a 64-bit reg) so only uint64 triggers on the LP64 host.
- **Fix shape**: Dispatch int↔float on srcTyp.Signed, mirroring the existing isUnsigned gate in lowerBinOp/emitBinop. VM: add a BC_UITOF opcode (vm.bni) doing cast(float64, cast(uint64, ...)); lower_cast.bn selects it when srcTyp.IsInteger()&&!srcTyp.Signed. aarch64: add Ucvtf (encoding 0x9E630000, sibling of Scvtf 0x9E620000) and Fcvtzu (0x9E790000, sibling of Fcvtzs 0x9E780000) to asm/aarch64/aarch64_fp.bn, then gate emitFloatCast on signedness. x64: add the uint64→double add-back idiom (no native instr) — test sign bit, if set: shr+or, Cvtsi2sd, addsd self; else plain Cvtsi2sd — plus Cvttsd2usi (the saturating-unsigned float→int idiom) for the reverse; gate on signedness. Add a shared signedness predicate so VM and natives can't diverge. Also fix the symmetric float→int unsigned path (same gate).
- **Files**: `pkg/binate/vm/lower_cast.bn (int→float arm:38-43; float→int:44-49)`; `pkg/binate/vm/vm_exec_helpers.bn (BC_SITOF:195-210; BC_FTOSI:211-223)`; `pkg/binate/vm.bni (add BC_UITOF / BC_FTOUI opcodes)`; `pkg/binate/native/aarch64/aarch64_float.bn (emitFloatCast:179-227)`; `pkg/binate/asm/aarch64/aarch64_fp.bn (add Ucvtf, Fcvtzu)`; `pkg/binate/native/x64/x64_float.bn (emitFloatCast:220-270)`; `pkg/binate/asm/x64/x64_fp.bn (add unsigned cvt support)`
- **Tests**: covered: conformance/matrix/scalar/int-to-float/64/unsigned — xfailed in builder-comp-int, builder-comp-int-int, builder-comp-comp-int, and all three native lanes; /32/unsigned passes as the zero-extended baseline. GAP: no float→int UNSIGNED cell (the symmetric defect); add a float-to-uint cell to the scalar matrix gen, and an int↔float ROUND-TRIP cell for the uint64 high-bit value.

### 4.5 Float-literal converter 1 ULP low for ~38+ sig-digit literals just above a binary tie (round-bit loss in the 128-bit sticky-collapse)

**MAJOR · CONFIRMED · UNBLOCKED 2026-06-05 (BUILDER compiles `math/big`)**

- **Root cause**: common.ParseFloatLitToBits holds the significand in a 128-bit (hi:lo) window and collapses everything below the kept 53 bits into a single sticky flag (packRounded, common_float.bn:182-268 uses roundBit + sticky). For ~38+ significant digits sitting JUST ABOVE a rounding tie, the exact round bit is lost and it rounds DOWN — 1 ULP low. Confirmed vs strconv + big.Float (~50% of constructed just-above-tie inputs diverge, all +1 ULP in strconv's favor). The file's own header comment (common_float.bn:13-18) claims it 'reproduces strconv.ParseFloat exactly across the full f64 range' — that comment is WRONG for the table-maker's-dilemma tail and should be corrected. LLVM (its own strtod) is correct; VM + natives share this converter so all three are wrong identically. Realistic literals (≤~37 sig digits) are correct.
- **Fix shape**: Proper fix is exact rounding via `pkg/std/math/big` (mantInt*10^exp as a Nat, extract 53 bits + round-to-even from the exact remainder — Go's slow-path). **UNBLOCKED (verified 2026-06-05)**: the filed "cmd/bnc's BUILDER tree can't import stdlib big/strconv" caveat is STALE for `math/big` — the current BUILDER (`bnc-0.0.7`) compiles and runs a `pkg/std/math/big`-importing program correctly (`Nat.Mul` → 3000000). `math/big` is float-free integer big-num (no floats/generics/closures/interfaces), so it IS BUILDER-compilable; only `strconv`-as-a-whole stays blocked (its `ftoa.bn` uses floats), and this fix needs `math/big` directly, not `strconv`. So: in `pkg/binate/native/common`, `import "pkg/std/math/big"` and replace the 128-bit-window sticky-bit rounding in `ParseFloatLitToBits` with exact `Nat`-based mantInt*10^exp rounding. Remaining gate before landing: a layering check (confirm no tier-hygiene rule forbids the compiler tree depending on tier-1 stdlib — a layering question, not BUILDER-compilability). The interim 256-bit-window patch is no longer needed if the proper fix lands. Also correct the misleading 'reproduces exactly' comment to state the round-bit limitation.
- **Files**: `pkg/binate/native/common/common_float.bn (packRounded:182-268; ParseFloatLitToBits:336-433; header comment:13-18)`; `pkg/std/strconv (ParseFloat core — the canonical exact implementation to route through)`; `conformance/538_float_lit_tie_roundbit.bn`
- **Tests**: covered: conformance/538_float_lit_tie_roundbit — passes on C/LLVM, xfailed in builder-comp-int, builder-comp-int-int, builder-comp-comp-int, and all three native lanes (the literal 1.0000000000000001110223024625156540424 → expects 0x3FF0000000000001, converter yields ...000).
