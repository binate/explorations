# Code-Red P2 — Plan 4: Scalar &amp; float-literal value correctness (sub-word narrowing, 64-on-32, unsigned int↔float, float-const materialization, div/0 panic)

> One of four disjoint code-red P2 work plans (partition in `plan-code-red-p2.md`).
> Source-confirmed: each defect cites root cause, fix shape, files, and test status.
> Defect details are tracked in `claude-todo.md`; this plan is the execution view.

## Summary

All six defects share one root cause class: a scalar's observable value is defined by its resolved type (width + signedness), but the VM and both native backends compute in full host-word registers / always-signed FP conversions, never re-establishing the type's invariant. LLVM is correct for free (true-width SSA + isUnsigned-gated cvt). The unifying fix is plan-code-red.md §3.8's "narrow once in a shared classifier": (a) a post-op sub-word narrow keyed on result type, (b) signedness-dispatched int↔float (mirroring the cmp/div/shift selection that already exists), (c) float-const materialization at resolved width (F64BitsToF32Bits). The divide-by-zero work is the ratified-but-unimplemented OP_DIV_CHECK + unsafe_div/unsafe_rem, mirroring OP_BOUNDS_CHECK/unsafe_index exactly. Crucially, the two DEFERRED float-const blockers (539/844 float32, and the 538/777 round-bit) were filed as "blocked on a new BUILDER release" — I confirmed BUILDER is now at bnc-0.0.7, cut 2026-06-04 from a tree that already contains common.F64BitsToF32Bits (commit 8042d609, 2026-06-03, is an ancestor of the 0.0.7 bump d407a66a). So the 539 float32-const fix is very likely UNBLOCKED now and should be re-attempted, not left deferred.

## Unifying strategy

Route every coercion through a small set of type-driven classifiers, one per concern, consulted on all three non-LLVM targets identically (LLVM stays correct): (1) a "narrow sub-word result" step applied after every value-producing integer op, keyed on instr.Typ.Width/Signed — implemented per-target (VM: a BC_NARROW with width/signedness carried in BCInstr.Imm/Aux, emitted by the lowerer right after the arith op, OR a fused narrow inside execArithOp using a width field added to BCInstr; natives: a sext/zext/and-mask tail after Add/Mul/etc. gated on Width<wordbits). (2) Int↔float signedness dispatch: add the unsigned op family (VM BC_UITOF; aarch64 Ucvtf/Fcvtzu — NOT YET in the asm lib, must be added; x64 the uint64→double add-back idiom since Cvtsi2sd has no unsigned form) selected by srcTyp.Signed, exactly mirroring lowerBinOp/emitBinop's existing isUnsigned gate. (3) Float-const at resolved width: route emitConstFloat / OP_CONST_FLOAT through common.F64BitsToF32Bits when the const type is float32. (4) Divide-by-zero: one OP_DIV_CHECK emitted in genBinary + the compound-assign path before OP_DIV/OP_REM, lowering to an rt.DivCheck call on every backend, with unsafe_div/unsafe_rem builtins passing checked=false. The decisive structural point: BCInstr (vm.bni:342) and ins.Typ (natives) both already carry — or can carry — the result type, so the classifier has its input at every site; the failure was simply that no narrowing/unsigned step consulted it.

### 4.5 Float-literal converter 1 ULP low for ~38+ sig-digit literals just above a binary tie (round-bit loss in the 128-bit sticky-collapse)

**MAJOR · CONFIRMED · FIXED/LANDED 2026-06-06 (binate 58570970) — via strconv, not math/big**

- **Root cause**: common.ParseFloatLitToBits held the significand in a 128-bit (hi:lo) window and collapsed everything below the kept 53 bits into a single sticky flag. For ~38+ significant digits sitting JUST ABOVE a rounding tie, the exact round bit was lost and it rounded DOWN — 1 ULP low. LLVM (its own strtod) is correct; the VM + natives shared this converter so all three were wrong identically. Realistic literals (≤~37 sig digits) were correct.
- **Fix (landed 58570970)**: ParseFloatLitToBits now delegates to `pkg/std/strconv.ParseFloat` (the user's call: route through strconv, not a hand-rolled math/big path) and `bit_cast`s its float64 result to the IEEE pattern. strconv.ParseFloat evaluates the literal as an exact rational over `big.Nat` (round-half-to-even, free of the table-maker's dilemma); the rounding is integer arithmetic and only the final bits are reinterpreted, so the value stays identical across LLVM-/VM-/native-built bnc. The ~300-line 128-bit-window machinery (packRounded et al.) is deleted; F64BitsToF32Bits (the independent float32 narrowing) is unchanged. **Layering/BUILDER gate resolved empirically**: this makes cmd/bnc depend on the real stdlib (strconv→math/big→errors) for the first time, and the current BUILDER (bnc-0.0.7) compiles it cleanly — gen1 + gen2 (builder-comp-comp-int) both build. So the plan's earlier "strconv-as-a-whole stays blocked (its ftoa.bn uses floats)" caveat was STALE.
- **Files (as landed)**: `pkg/binate/native/common/common_float.bn (ParseFloatLitToBits → strconv; converter helpers removed)`; `pkg/binate/native/common/common_float_test.bn (comments + a round-bit unit guard)`; `conformance/538_float_lit_tie_roundbit.bn (comment)`; dropped the 538 + matrix/const/*/float64/tie xfails on the five runnable lanes.
- **Tests**: `538_float_lit_tie_roundbit` and the seven `matrix/const/*/float64/tie` cells XPASS→pass on every runnable lane (builder-comp-int, -int-int, -comp-int, native aa64, native x64_darwin) — 40 xfails dropped; a new `TestParseFloatRoundBit` unit guard pins the same literal. ### 4.6 Divide-by-zero / mod-by-zero must panic (DEFINED) + unsafe_div/unsafe_rem builtins — RATIFIED, implementation not started

**MAJOR · CONFIRMED RATIFIED (contract in ir.bni; OP_DIV_CHECK + builtins NOT implemented; coverage GAP)**

- **Root cause**: Integer / and % are accidentally-UB across backends today: LLVM sdiv/udiv (UB at /0), native IDIV/SDIV (SIGFPE only by hardware accident), VM host /,% (vm_exec_pure.bn:22-43 — BC_DIV/REM/UDIV/UREM do raw regs[Src1]/regs[Src2] with no zero guard). The 2026-06-04 decision (binate 699f08eb, documented in ir.bni:30-38 OP_DIV/OP_REM) ratifies: a zero divisor OR signed-MIN/-1 overflow is a DEFINED panic on all four backends (matches Go/Rust). There is NO OP_DIV_CHECK op, no rt.DivCheck runtime fn, and no unsafe_div/unsafe_rem builtins (grep confirms only the ir.bni doc-contract exists). The model OP_BOUNDS_CHECK lowers to a call @bn_pkg__builtins__rt__BoundsCheck (emit_instr.bn:397-407) → rt.BoundsCheck → rt.BoundsFail (rt.bn:210-215); unsafe_index opts out by passing checked=false to genIndex (gen_access.bn:32-45).
