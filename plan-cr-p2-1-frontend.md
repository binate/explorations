# Code-Red P2 — Plan 1: Front-end correctness: const materialization, float32 coercion, address-of slice element, declaration resolution, int-int loader

> One of four disjoint code-red P2 work plans (partition in `plan-code-red-p2.md`).
> Source-confirmed: each defect cites root cause, fix shape, files, and test status.
> Defect details are tracked in `claude-todo.md`; this plan is the execution view.

## Summary

Six confirmed front-end defects, all rooted in IR-gen / type-checker logic that handles the integer/literal/struct-element common case but silently drops or mis-handles the adjacent cases: untyped-float narrowing at call-arg/field/return (works only at var-init/named-const because only those route through ensureWidth); local consts (genDecl handles only DECL_VAR, so a local const never materializes and reads as 0); non-int const-EXPRESSIONS and const-as-array-dimension (evalConstExpr / classifyConstLit / resolveTypeExpr / checker evalConstInt are all integer-and-literal-only); &slice[i] for scalar-element slices (genIndexPtr's slice arms are gated on TYP_STRUCT, so scalar slices fall through to the r-value load+inttoptr path); untyped single const forward-ref (collectDecls forward-registers only typed consts); and the int-int loader rt-not-found (a default-mode red whose root cause is genuinely unknown). The theme: the front end's "easy path" silently materializes 0 / drops narrowing / wild-pointers instead of doing the right thing, and the fix in most cases is to route the missing shape through the same coercion/materialization machinery the working sibling already uses.

## Unifying strategy

Two structural threads tie most of these together. (1) FLOAT/WIDTH COERCION: the var-init path narrows correctly because gen_stmt.bn:249 calls ensureWidth unconditionally; arg/field/return do not. Extend the SAME width-coercion (ensureWidth, which already lowers untyped-float→float32 to fptrunc — verified emit_ops.bn:231) to the call-arg (coerceArg, gen_call.bn:41), composite-field (gen_composite.bn:140 store), and return (gen_return.bn:98-103, which today gates ensureWidth on TYP_UNTYPED_INT only) sites. Prefer adding/using one shared helper (e.g. coerceScalarWidth) called at all four scalar-store sites so a future width gap can't reopen at one site. (2) CONST MATERIALIZATION / EVALUATION: local-const, non-int const-expr, and const-as-array-dim all stem from the const value-evaluation machinery being integer-and-literal-only (evalConstExpr int-only; classifyConstLit literal-only; checker defineConst attaches type-but-not-value for non-int; evalConstInt returns 0 for const idents). A principled fix extends const evaluation to (a) materialize local consts (route local DECL_CONST through the moduleConsts read-path or a local-const slot), (b) fold non-int const-EXPRESSIONS at the right type, and (c) resolve const idents/exprs in the array-dimension path (both resolveTypeExpr's TEXPR_ARRAY and the checker's evalConstInt). The address-of and forward-ref and loader items are independent point fixes. NOTE: do NOT change language semantics — if any const-expr form is to be rejected rather than evaluated, that is a clear diagnostic, not silent 0, and either way the open iota-repeat-vs-plain question (below) must be settled by the user before asserting group-member behavior.

## Defects

### 1.1 A float literal narrowed to float32 is NOT coerced at call-arg / composite-field / return positions

**CRITICAL · CONFIRMED**

- **Root cause**: The var-init path narrows via gen_stmt.bn:249 `val = ensureWidth(ctx, b, val, typ)` (unconditional); the other three positions do not. (a) CALL-ARG: gen_call.bn:204-206 calls genExprOrFuncRef then coerceArg; genExprOrFuncRef (gen_util.bn:135-155) narrows INT literals only (isTypedInt/needsHintNarrowing) — no float branch; coerceArg (gen_call.bn:41-94) handles string/nil/managed-slice/struct-copy/@Iface but never calls ensureWidth and has no float-narrowing. So `take(0.1)` keeps val.Typ=TYP_UNTYPED_FLOAT (double) into a float32 param. (b) FIELD: gen_composite.bn:50-59 calls genExprOrFuncRef then stores at :140 `b.EmitStore(fieldPtr, val)` with no width coercion — a float32 field gets a double-typed value. (c) RETURN: gen_return.bn:99-103 calls ensureWidth but GATES it on `val.Typ.Kind == TYP_UNTYPED_INT && retTyp.IsInteger()` — integer-only; an untyped-float return at a float32 slot is not narrowed, emitting `ret float <double>` (invalid LLVM IR). Verified fptrunc is the correct lowering once the value carries float32 type: emit_instr.bn:48-64 (const at float32 → fptrunc) and emit_ops.bn:228-238 (EmitCast double→float32 → fptrunc).
- **Fix shape**: Route arg/field/return scalar values through the same width-coercion the var-init path uses. Cleanest: add a shared `coerceScalarWidth(ctx, b, val, targetTyp)` (wrapping/extending ensureWidth so it covers the untyped-float→float32 case — ensureWidth already does because typeWidth(TYP_UNTYPED_FLOAT)=64 vs typeWidth(float32)=32, gen_binary.bn:200-219) and call it at: coerceArg (gen_call.bn:41, after the existing conversions, for scalar params), the composite-field store (gen_composite.bn just before :140 EmitStore, for scalar fields), and the return path (gen_return.bn — replace the TYP_UNTYPED_INT-only gate at :99-103 with a width-coercion that also fires for untyped-float→float). Be careful to only coerce scalars (don't disturb slices/structs/iface/funcvalue, which have their own conversions). CALIBRATION: this is an LLVM-correct fix; the VM/native float32 backend value bug (Class 5, 539/844 DEFERRED) is separate — so the {call-arg,field,return}/float32 cells should be unxfailed to match whichever working sibling they resemble (likely var-init/float32, which passes ALL modes, since the value is a literal not a named const; confirm by running, do NOT blanket-unxfail).
- **Files**: `pkg/binate/ir/gen_call.bn (coerceArg ~:41-94, and the arg loop ~:204-206)`; `pkg/binate/ir/gen_composite.bn (keyed/positional field stores ~:50-140)`; `pkg/binate/ir/gen_return.bn (~:98-103 the int-only ensureWidth gate)`; `pkg/binate/ir/gen_binary.bn (ensureWidth ~:214 / typeWidth ~:200, if a shared helper is added here)`; `pkg/binate/ir/gen_util.bn (genExprOrFuncRef ~:107, if float-hint narrowing is added there instead)`
- **Tests**: covered: conformance/matrix/const/{call-arg,field,return}/float32/{tenth,neg,half}.bn (9 cells), all xfailed in all default modes with reason 'float literal not narrowed to float32 at this read-form'. After the fix, recalibrate the xfail set to match the working var-init/named-const float32 sibling (var-init passes all modes; named-const is xfailed on the 3 native + 3 int modes for the deferred backend value bug).

### 1.2 Local const declarations silently materialize 0

**CRITICAL · CONFIRMED**

- **Root cause**: genDecl (gen_stmt.bn:125-333) handles ONLY `d.Kind == ast.DECL_VAR`. A local `const C T = V` reaches genStmt's STMT_DECL arm (gen_stmt.bn:62-67 → genDecl) but, being DECL_CONST not DECL_VAR, falls through genDecl doing NOTHING: no alloca, no moduleConsts entry, no local-const slot. The subsequent read of `C` in genExprInner's EXPR_IDENT path (gen_expr.bn:60-87) finds neither a var (lookupVar nil) nor a moduleConst (only top-level genConst/genConstGroup populate moduleConsts), so it returns the placeholder `b.EmitConstInt(0, types.TypInt())` at gen_expr.bn:86. Value V is dropped → reads as 0, type-independent. Package consts, group members, inline literals all work; only the local const form is broken.
- **Fix shape**: Bind the local const's materialized value at its read site, mirroring the package-const path. Concrete options (pick after a quick look at how moduleConsts is scoped): (a) handle DECL_CONST in genDecl by registering the const in a local/function-scoped const table that the EXPR_IDENT read path consults before falling through to the 0-placeholder — reusing classifyConstLit + evalConstExpr to compute the value/Kind exactly as genConst does; OR (b) materialize the const into an alloca'd slot at decl point (like a var) and defineVar it so reads load it. Option (a) is closer to the existing const model and avoids an unnecessary slot. Either way it must support all scalar types (int widths, float32/64, bool, char) — route through the same classifyConstLit/evalConstExpr/EmitConstFloat/EmitConstBool dispatch the package-const read uses (gen_expr.bn:71-84). The checker already accepts local consts (defineConst in the function scope), so this is purely an IR-gen materialization gap. (Rejecting local const at type-check is the documented fallback ONLY if local const is intentionally unsupported — but it is valid Binate, so materialize it.)
- **Files**: `pkg/binate/ir/gen_stmt.bn (genDecl ~:125 — add DECL_CONST handling, or route STMT_DECL of a const)`; `pkg/binate/ir/gen_expr.bn (EXPR_IDENT const read ~:60-87 — consult local-const table before the 0 placeholder)`; `pkg/binate/ir/gen_const.bn (reuse classifyConstLit/evalConstExpr ~:207/:66 for the value computation)`
- **Tests**: covered: conformance/matrix/const/local-const/{float32,float64,int16,int64,uint32,uint64,uint8}/{tenth/neg/half or min/max/...} (the README cites 12 cells), all xfailed in all default modes. Unxfail after fix. (Note: the bug is type-independent, so one representative cell would suffice per the matrix-vs-regressions note, but the cells already exist.)

### 1.3 Non-integer const-EXPRESSIONS (binary float, bool comparison) and const-as-array-dimension are dropped (read as int 0 / wrong length)

**MAJOR · CONFIRMED**

- **Root cause**: Three manifestations, one family: (1) BINARY FLOAT const-expr (`const X float64 = 1.5 + 2.5`): IR-gen's classifyConstLit (gen_const.bn:207-223) matches only a BARE/unary-minus float or bool LITERAL; an EXPR_BINARY initializer falls through to the integer-only evalConstExpr (gen_const.bn:66-167), which can't evaluate float ops, so genConst drops the const (gen_const.bn:245 `if !ok { return }`) → read falls to EmitConstInt(0) at gen_expr.bn:86. (2) BOOL COMPARISON (`const B bool = 1 < 2`): same path — evalConstExpr has no comparison ops, drops the const → reads 0/false. The checker accepts these (checkConstDecl, check_decl.bn:409-428, defines the const with its type via defineConst but attaches NO value — the HasLitVal/LitMag machinery, makeUntypedIntWithLit at check_decl.bn:435, is integer-only). (3) CONST-AS-ARRAY-DIMENSION (`const N int = 3; var a [N]int`): resolveTypeExpr's TEXPR_ARRAY arm (gen_util.bn:354-359) does `parseIntLit(te.Len.Name)` on the raw ident text 'N' (garbage; observed 30), never resolving the const; AND the checker's evalConstInt (checker_util.bn:79-96) returns 0 for an EXPR_IDENT that is SYM_CONST (line 88-90) and ERRORS ('array length must be a constant integer') for an EXPR_BINARY like `[N+1]int` (line 94) even though it is a constant.
- **Fix shape**: Extend const evaluation to non-int expressions at the correct type, and resolve consts in the array-dim path. Concretely: (a) IR-gen — fold float const-EXPRESSIONS at float precision and bool const-comparisons to a bool (either a typed const-expr evaluator distinct from the integer evalConstExpr, or extend classifyConstLit/genConst to recognize EXPR_BINARY/EXPR_UNARY over float/bool operands and emit the folded value via EmitConstFloat/EmitConstBool). (b) ARRAY-DIM IR-gen — in resolveTypeExpr TEXPR_ARRAY (gen_util.bn:354-359), when te.Len is an EXPR_IDENT/EXPR_BINARY, evaluate it via evalConstExpr (which already resolves const idents through lookupConst) instead of parseIntLit on the raw text. (c) ARRAY-DIM checker — in evalConstInt (checker_util.bn:79-96), actually READ the const's value for a SYM_CONST ident (don't return 0), and evaluate constant binary expressions (`N+1`) rather than erroring. Fallback per todo: if any const-expr form is deliberately unsupported, emit a CLEAR diagnostic, never silent 0. OPEN LANGUAGE QUESTION — FLAG, DO NOT ASSUME: a BARE const-group member currently takes PLAIN iota (gen_const.bn:293-299), not Go-style repeat-previous-expression, so `const ( B0 int = 1 << iota; B1; B2 )` gives plain-iota values not the 1,2,4,8 bit-flag idiom. Whether to implement repeat-previous is a SPEC decision the user must make before a group-member const-expr cell can assert behavior. Surface this to the user; do not pick a side.
- **Files**: `pkg/binate/ir/gen_const.bn (evalConstExpr ~:66, classifyConstLit ~:207, genConst ~:231, genConstGroup ~:258 incl. the iota-repeat question ~:293-299)`; `pkg/binate/ir/gen_util.bn (resolveTypeExpr TEXPR_ARRAY ~:354-359)`; `pkg/binate/types/checker_util.bn (evalConstInt ~:79-96)`; `pkg/binate/types/check_decl.bn (checkConstDecl ~:383, defineConst value-attachment for non-int)`; `pkg/binate/types/check_expr_constfold.bn (foldIntArith/foldIntBitwise ~:25/:84 — model for a float/bool fold extension)`
- **Tests**: covered (pinning xfails): conformance/regressions/const-expr/{float-binary-add,float-binary-div,float-binary-mul,bool-comparison,array-dim}.bn all xfailed in all default modes; green baselines confirm the int/literal paths fold: {int-arith,int-bitwise,int-paren,int-of-const,float-neg-literal,bool-literal}.bn. GAP: no cell asserts group-member const-expr behavior pending the iota-repeat spec decision; do not add one until decided.

### 1.4 &slice[i] (address-of a slice element) lowers to a wild pointer for scalar-element slices

**CRITICAL · CONFIRMED**

- **Root cause**: genIndexPtr (gen_access.bn:84-142) returns a proper element GEP for: IDENT arrays (TYP_ARRAY → EmitGetElemPtr, :98), IDENT pointers (:105), and slices-of-STRUCT (IDENT :108-113 and SELECTOR :132-138, via EmitSliceElemPtr) — but its slice arms are GATED on `arrTyp.Elem.Kind == types.TYP_STRUCT`. For a SCALAR-element slice (`@[]uint8`, `*[]uint8`), neither slice arm matches, so genIndexPtr returns nil. Back in genUnary's address-of (gen_expr.bn:152-158), a nil from genIndexPtr falls through to `return genExpr(ctx, b, e.X)` (gen_expr.bn:170), which lowers `s[i]` as an R-VALUE: genIndex → EmitSliceGet(collection, index, elemTyp) (gen_access.bn:80) returns the loaded element VALUE (a uint8). The `&` then hands back that byte, which downstream is used as a pointer (LLVM inttoptr / native), producing a wild pointer → SIGSEGV on write-through. Fixed arrays are unaffected (they take the GEP arm). Crashes identically in bnc and bni → shared-IR defect, not a backend. The correct primitive (EmitSliceElemPtr) already exists and is used by the struct arm.
- **Fix shape**: Remove the TYP_STRUCT-only restriction on genIndexPtr's slice arms so &s[i] yields the GEP'd element address for ANY element type. Concretely: in the IDENT branch (gen_access.bn:108-114) and the SELECTOR branch (:132-139), change the guard from `isSliceType(arrTyp) && arrTyp.Elem != nil && arrTyp.Elem.Kind == TYP_STRUCT` to `isSliceType(arrTyp) && arrTyp.Elem != nil` (drop the .Kind==TYP_STRUCT clause), keeping the EmitSliceLen bounds-check + EmitSliceElemPtr emission identical. This mirrors the fixed-array address path. Verify the bounds check stays (currently present for the struct arm). For the SELECTOR scalar case the existing `genExpr(ctx, b, e.X)` load of the slice header is fine for EmitSliceElemPtr. No language-semantic change: arrays already support &a[i] and raw pointers are the documented hot-path escape, so emitting the address is the intended behavior.
- **Files**: `pkg/binate/ir/gen_access.bn (genIndexPtr slice arms ~:108-114 IDENT, ~:132-139 SELECTOR)`; `pkg/binate/ir/gen_expr.bn (genUnary address-of fall-through ~:152-170 — verify behavior once genIndexPtr returns non-nil)`
- **Tests**: covered: conformance/599_addr_of_slice_elem.bn — write-through + read-back on @[]uint8 and *[]uint8 (expected 99/88), xfailed in all 6 default modes. Unxfail after fix. GAP to consider: a managed-slice-of-struct &s[i].field shape and a SELECTOR-based scalar slice (`c.Items[i]` where Items is @[]uint8) to exercise the SELECTOR arm; add if the worker wants the SELECTOR scalar path pinned independently.

### 1.5 Untyped single const (const X = 5) is not forward-referenceable

**MAJOR · CONFIRMED**

- **Root cause**: collectDecls (check_decl.bn:52-125) forward-registers a single DECL_CONST only when BOTH `d.TypeRef != nil && d.Value != nil` (line 87-91). An untyped single const (`const X = 5`, TypeRef==nil) is NOT registered in this collection pass, so a decl checked BEFORE it (a forward ref, or a sibling file ordered ahead — package files are merged) sees X as `undefined`. Typed `const X int = 5` is fine (registered at :88-90). The DECL_GROUP arm (:101-113) already handles the bare untyped group member with an untyped-int placeholder (the fixed case, binate 88c9c0b7), but the single untyped case was left because it is harder: an untyped single const's type depends on its value, and naively checkExpr-ing the value during collection would emit spurious `undefined` for reference-valued consts (`const X = Y; const Y = 5`).
- **Fix shape**: In collectDecls' single DECL_CONST path (check_decl.bn:86-92), forward-register an untyped single const when its value is a SIMPLE LITERAL (int/string/float/bool/char) whose type is unambiguous and dependency-free — inferring the placeholder type from the literal kind (mirror resolveGlobalVarType's literal→type mapping). Leave reference/expression-valued untyped consts for the existing later full-check pass (do NOT checkExpr them during collection) to avoid the spurious-undefined trap. This is the loud-failure (compile-time `undefined`), not a silent miscompile, so lower urgency than the silent items — but it is the same collectDecls structural gap as the (fixed) group case.
- **Files**: `pkg/binate/types/check_decl.bn (collectDecls single DECL_CONST arm ~:86-92)`
- **Tests**: covered (probe asserting buggy behavior): pkg/binate/types/check_decl_test.bn TestForwardRefUntypedSingleConstKnownGap (~:294-301) currently asserts expectError(c); FLIP to expectNoErrors when fixed. GAP: add a conformance test mirroring 526_forward_ref_iota_const for the single-const case (per the todo's stated fix plan).

### 1.6 136_grouped_imports / 383_cross_pkg_iface_dtor: package pkg/builtins/rt not found under int-int (loader resolution at double-VM depth)

**MAJOR · CONFIRMED (symptom); root cause unknown — needs investigation**

- **Root cause**: On builder-comp-int-int (the double-VM default mode) ONLY, both multi-package tests fail at compile time with `package "pkg/builtins/rt" not found`; green on builder-comp-int and builder-comp-comp-int and all other modes. The error originates in loader.bn:236-244 (both BniPath and ImplPath search loops missed). The loader's search paths are seeded from NewLoader(root) (loader.bn:14-27, AddRoot → BniPath+ImplPath) plus applyPathFlags/addLoaderPaths (cmd/bni/util.bn:154-173). At int-int depth — bni runs bnc, which compiles a program that is itself run under a second bni — the `root` / search-path setup for the transitively-imported core package pkg/builtins/rt is not propagated to the inner level. Confirmed pre-existing on clean trees (17c722d1 / pre-float-fix), so it is a main-branch loader-path regression, not from any @func/@Iface work. Only certain multi-package tests trip it (grouped-imports + cross-package), suggesting the missed resolution is a transitive core import reached only through a particular import shape. Genuine investigation item — do not point-patch the loader's not-found branch.
- **Fix shape**: INVESTIGATE how pkg/builtins/rt's search path is established at each interpreter nesting depth and why int-int loses it. Steps: (1) instrument/trace the loader's BniPath/ImplPath contents at the inner (int-int) invocation vs the working comp-int/single-int invocations to see which root is missing; (2) trace how the conformance run.sh int-int mode invokes the inner bni and what root/-I/-L it passes (the inner bni's NewLoader root may default to a cwd/relative root that doesn't contain pkg/builtins/, vs the outer level which gets it from the toolchain root); (3) fix the path propagation so every nesting depth resolves transitive core imports identically (the Class-8 invariant: identical resolution at single-int / comp-int / int-int). The fix likely lives in the int-int harness wiring (how the inner bni's root/search-paths are derived), not the loader library itself. Until root-caused, at minimum add .xfail.builder-comp-int-int markers to 136_grouped_imports and 383_cross_pkg_iface_dtor (they currently have NONE, so the mode is silently red) with a one-line reason pointing at this todo — but the real deliverable is the fix, not the xfail.
- **Files**: `pkg/binate/loader/loader.bn (loadPackage search loops ~:164-244, NewLoader/AddRoot ~:14-35)`; `cmd/bni/util.bn (loadBuiltinBNIs ~:67-85, addLoaderPaths ~:167, applyPathFlags ~:158)`; `cmd/bni/main.bn (loader root derivation — how NewLoader's root is chosen)`; `conformance/run.sh (int-int mode inner-bni invocation: root / -I / -L args)`; `conformance/136_grouped_imports/ and conformance/383_cross_pkg_iface_dtor/ (add .xfail.builder-comp-int-int if not fixed first)`
- **Tests**: covered (currently RED, unmarked): conformance/136_grouped_imports + conformance/383_cross_pkg_iface_dtor fail only in builder-comp-int-int with no xfail markers (default-mode red, masking coverage). Either fix (then they go green) or add the int-int xfail markers as an interim. GAP: a minimal 2-package int-int reproducer that imports pkg/builtins/rt transitively would isolate the path-propagation bug faster than the two existing larger tests.

## Sequencing

Recommended order (independent point fixes first, structural threads grouped): (1) &slice[i] address-of (CRITICAL, smallest, fully isolated — drop the TYP_STRUCT guard in genIndexPtr; unxfail 599). (2) Local const materialization (CRITICAL; self-contained in genDecl + the EXPR_IDENT read path; reuses existing classifyConstLit/evalConstExpr). (3) float32 coercion at call-arg/field/return (CRITICAL; do this as ONE change introducing/using a shared coerceScalarWidth helper at all three sites, so the three matrix sub-cells fix together; recalibrate xfails to the var-init sibling profile). (4) Non-int const-expr + array-dim (MAJOR; touches both IR-gen and checker; do array-dim and float/bool const-expr together since they share the const-evaluation-is-int-only root). FLAG the iota-repeat-vs-plain spec question to the user BEFORE asserting any group-member const-expr behavior. (5) Untyped single-const forward-ref (MAJOR, loud-fail; small collectDecls change; flip the probe test, add conformance). (6) int-int loader rt-not-found (MAJOR; investigation-heavy, root cause unknown — start with tracing, may overlap with harness wiring; can run in parallel with the others since it touches loader/harness, not IR-gen/checker). Dependencies: items 2 and 4 both touch gen_const.bn (classifyConstLit/evalConstExpr) — do 2 first, then 4 builds on the same helpers. Items 3 and 4 both involve scalar coercion but at different layers (3 = float width at store sites; 4 = const value evaluation) — no hard ordering, but landing 3's shared coerceScalarWidth first gives 4 a clean place to hook array-dim const width if needed.

(C) gen_composite.bn — this plan adds scalar field width coercion near the :140 store; the refcount plan edits the managed-field acquire arms (:114-138) just above.
