# Code-Red-2 — Plan 3: Backends (native aa64/x64/arm32 + bytecode VM)

> One of three disjoint Code-Red-2 fix plans (mirrors the P2 partition in `plan-cr-p2-1-frontend.md` / `plan-cr-p2-3-abi.md`).
> Source-confirmed against `temp-binate-4/pkg/binate/native/{aarch64,x64,common}` and `pkg/binate/vm` (READ-ONLY; line numbers are what the current tree shows). Defects of record live in `claude-todo.md`; this is the execution view.

## Theme

Every defect here is one unfilled cell of the FEATURE × MODIFIER × POSITION product, and all six fail the two Code-Red-2 invariants directly. PATH PARITY: the sub-word-narrow re-reduction, the eightbyte-coalescing multi-return pack/collect, and the small-aggregate funcval-return convention each got fixed in ONE lowering path (the binary-op path, the *direct*-call path) but not in the parallel paths (the unary-op path, the iface-method and func-value collect paths, the small-aggregate shim) — so a value of the exact triggering shape silently breaks where its sibling passes. WRAPPER TRANSPARENCY shows up in the VM's BC_EXTRACT, which reads a full host word regardless of the field's sub-word type, and in the global-materialization parity work that must peel the same `TYP_READONLY` / named wrappers codegen's global emitter now peels. The fixes are confined to the backend packing/narrowing/materialization sites; none touches `pkg/binate/ir`, `pkg/binate/types`, or `pkg/binate/codegen`.

## Defects

### Defect: native `emitUnop` (OP_NEG / OP_BITNOT) omits the sub-word re-narrow that `emitBinop` performs (PATH-PARITY gap)

**Symptom**: a sub-word unary result (`~x` / `-x` for `uint8/16/32`, `int8/16/32`) leaves dirty upper bits in the host register on aa64 and x64. Latent for most consumers because the next sub-word op / compare / cast re-narrows; exposed when the un-narrowed value is consumed directly by a widening read, an unsigned compare, a shift count, or a store of the full word.

**Root cause (confirmed)**: aa64 `aarch64_ops.bn` — `emitBinop` ends with `emitSubWordNarrow(a, rd, ins.Typ)` at line 123 before `return true`. `emitUnop` (lines 187-214) emits `Neg` (line 201) / `Mvn` (line 205) and immediately `return true` with NO `emitSubWordNarrow` call. x64 `x64_ops.bn` — identical asymmetry: `emitBinop` calls `emitSubWordNarrow(a, rd, ins.Typ)` at line 156, and `emitDivOrRem` at line 217, but `emitUnop` (lines 340-372) emits `Neg` (line 359) / `Not` (line 363) and returns with no narrow. The shared classifier `common.SubWordNarrow` and the per-backend `emitSubWordNarrow` (aa64 lines 20-34, x64 lines 29-45) already exist and are no-ops for host-word / non-integer types, so the call is safe to add unconditionally. (OP_NOT — the bool path, aa64 line 209 / x64 line 367 — needs no narrow: it XORs bit 0 of a 1-bit value.)

**Fix shape**: append `emitSubWordNarrow(a, rd, ins.Typ)` immediately before the `OP_NEG` and `OP_BITNOT` `return true`s in both `emitUnop`s — mirroring `emitBinop` exactly. Cleanest: narrow once just before the final `return true` for the NEG/BITNOT arms (skip the NOT arm). `~` on a sub-word is the most exposed: `~x` over a `uint8` should yield an 8-bit complement; without the uxtb the high bits carry the host-width complement.

**Files/functions**: `pkg/binate/native/aarch64/aarch64_ops.bn` (`emitUnop`, the OP_NEG arm ~line 200-203 and OP_BITNOT arm ~line 204-207); `pkg/binate/native/x64/x64_ops.bn` (`emitUnop`, OP_NEG arm ~line 358-361 and OP_BITNOT arm ~line 362-365). Reuses the existing `emitSubWordNarrow` in each file; touches NO shared/common file.

**Test coverage**: the matrix cell `conformance/matrix/scalar-diff` covers sub-word arith on the natives but, per `claude-todo.md` (`aa64-subword`, lines ~904-924), the unary `~`/`-` directly-consumed cell is NOT pinned — the `bitwise/not` cells exercise `~` but through LLVM/IR-gen typing (the `bitnot-result-type` fix), not the native re-narrow. GAP to fill: a probe that consumes a sub-word `~`/`-` result WITHOUT re-narrowing — e.g. `var y uint16 = ~x; print(cast(uint64, y))` or `if (~x) > k { … }` where the dirty high bits change the unsigned comparison — added as a `scalar-diff`-style cell or a native unit test in `aarch64_ops_test.bn` / `x64_ops_test.bn` asserting the uxt/sxt (or movzx/movsx) is emitted after NEG/BITNOT. The probe MUST avoid an intervening cast/compare that would mask the omission.

### Defect: aa64 mis-packs a sub-word / non-8-multiple struct returned through a FUNCTION-VALUE call (`funcval-return/five-u8`)

**Symptom**: `conformance/matrix/abi/funcval-return/five-u8` — a `@func() S` returning a 5×uint8 struct `{1,2,3,4,5}` prints `16,146,211,…` on native aa64 (and x64). The interface-dispatch sibling `iface-return/five-u8` and the DIRECT struct-return both pass; only the function-value-call result path mis-packs (`claude-todo.md` top entry, lines 33-36, marked CRITICAL/CONFIRMED).

**Root cause (confirmed)**: a small (≤ 8-byte) aggregate funcval return hits a CONVENTION MISMATCH between the caller's collect and the per-function shim. In `aarch64_call_indirect.bn`, `emitCallFuncValue` sets `aggregateRet` true for any non-multi-return `IsAggregateTyp(ins.Typ)` (lines 241-243) — a 5-byte struct qualifies — and then uses the **retbuf-in-X0** convention: it passes `X0 = SP + retbufOff` (lines 356-361) and after the call just produces the alloc-region pointer (lines 394-399), trusting the shim to have written the result through the retbuf. But the shim (`aarch64_funcvalue.bn` `emitFuncValueShims`) only builds a retbuf-aware body when `usePack` (`retSz > 8 && retSz <= 16`, line 213) or `useSret` (`retSz > 16`, line 212); a `retSz <= 8` aggregate falls into the **scalar/void shim** (lines 267-276), which ignores X0/retbuf, tail-branches, and lets the underlying return the packed struct in X0. The caller then reads an UNWRITTEN retbuf → garbage. (The `retWords = retSz/8; if retSz % 8 != 0 { retWords++ }` at lines 245-246 is a correct ceil and not the bug.) x64 is the same shape: `x64_call_indirect.bn` `aggregateRet` (lines 225-227) + retbuf-in-RDI (lines 284-292), with the x64 shim's `usePack` gated `retSz > 8` (line 183) so a ≤ 8-byte aggregate takes the scalar shim (lines 228-234).

**Fix shape**: close the missing convention arm for 1..8-byte aggregate funcval returns. Two equivalent routes; pick one and apply to BOTH backends symmetrically: (a) extend the shim's pack path to fire for `retSz >= 1 && retSz <= 16` (lower the `usePack` floor and have the ≤ 8-byte case store just RAX/X0 through retbuf) so the caller's existing retbuf convention is honored; OR (b) make `emitCallFuncValue` treat a ≤ 8-byte aggregate return as a register return (drop `aggregateRet` for `SizeOf <= 8`, collect RAX/X0 into the data region like the small-aggregate DIRECT-call path in `x64_call.bn:256-273` / `aarch64_call.bn:206-221`). Route (a) keeps the funcval ABI uniform (retbuf always meaningful for aggregates); route (b) keeps the shim minimal. Whichever: caller and shim MUST agree, and the choice must match how `emitFuncValueShims`'s `shimReturnSize` thresholds are read on the same backend. Confirm against the direct small-aggregate return, which already works.

**Files/functions**: `pkg/binate/native/aarch64/aarch64_funcvalue.bn` (`emitFuncValueShims`, the `usePack`/`useSret`/scalar split ~lines 211-276; `shimReturnSize` ~lines 133-138); `pkg/binate/native/aarch64/aarch64_call_indirect.bn` (`emitCallFuncValue`, `aggregateRet` ~241-245, retbuf setup ~356-364, collect ~394-399); `pkg/binate/native/x64/x64_funcvalue.bn` (`emitFuncValueShims` ~181-234; `shimReturnSize_x64` ~120-125); `pkg/binate/native/x64/x64_call_indirect.bn` (`emitCallFuncValue`, `aggregateRet` ~225-229, retbuf ~284-292, collect ~336-342).

**Test coverage**: pinned by `conformance/matrix/abi/funcval-return/five-u8` (xfail native aa64/x64, likely arm32) — flips to XPASS on fix. GAP: a 1-8-byte funcval return that is NOT sub-word-packed (e.g. `three-u16` = 6B, or a `{int32}` 4B) to prove the fix covers the whole ≤ 8 aggregate-return-via-funcval region, not just the five-u8 cell; plus a 9..16-byte funcval-return cross-check that the existing `usePack` path still passes (regression guard for route (a)).

The DIRECT-call sub-word multi-return was fixed (binate `b5616b32`) but the indirect collect sites were not updated to match.

**Root cause (confirmed)**: the callee now packs the tuple as a COALESCED SysV byte image in RAX (+RDX) — `x64_return.bn` `emitMultiReturnPack` (lines 155-196) materializes the tuple in the red zone at each field's `FieldOffset`, coalescing multiple sub-word fields into one eightbyte, then loads RAX/RDX. **Files/functions**: `pkg/binate/native/x64/x64_iface.bn` (`emitCallIfaceMethod` multi-return collect ~lines 191-208); `pkg/binate/native/x64/x64_call_indirect.bn` (`emitCallFuncValue` multi-return collect ~lines 345-362); optionally a new shared helper beside `x64_call.bn`'s `emitCall` collect (~lines 230-246) that all three call. ### Defect: VM mis-unpacks a sub-word (uint16) multi-return through interface dispatch — BC_EXTRACT reads a full host word (WRAPPER/width-TRANSPARENCY gap)

**Symptom**: `conformance/matrix/abi/iface-multi-return/u16/2` — a dispatched `mr() (uint16, uint16)` returning `(100, 200)` prints `13107300, 1` on the bytecode VM instead of `100, 200`. `13107300 = (200<<16)|100` — the first extracted component reads BOTH u16s out of word 0. The `int` variant is correct on the VM (`claude-todo.md` lines 27-30, CONFIRMED).

**Root cause (confirmed)**: the multi-return tuple is packed as a correct sub-word byte image, but `BC_EXTRACT` reads a full 8-byte word with no sub-word sizing. Pack side (`vm_exec_return.bn` `packMultiReturn`, lines 24-66): for a non-pair non-multiword scalar field it writes `var p *int = bit_cast(*int, dstBase + off); p[0] = regs[...]` (lines 52-55) — an 8-byte store at the field's `ResultOffsets[i]` (computed with natural alignment in `lower_func.bn:64-78`, so `(u16,u16)` → offsets 0 and 2). Reading the image back: lowering (`lower_instr.bn` OP_EXTRACT arm, lines 358-391) sets `bc.Imm = srcTyp.FieldOffset(instr.Index)` (line 369) and only marks pointer mode (`bc.Aux = 1`) for `isMultiWordField || isVMAddressAggregate` (lines 380-383) — a sub-word scalar field stays value-mode with `Aux == 0`. The exec (`vm_exec_helpers.bn` `BC_EXTRACT`, lines 430-447): the `Aux == 0` arm (lines 442-445) does `var p *int = bit_cast(*int, regs[Src1] + Imm); regs[Dst] = p[0]` — a FULL `*int` (8-byte) load at the byte offset, with no width mask. Extracting field 0 of the `(u16,u16)` image reads bytes[0:8] = `100 | (200<<16)` = 13107300 — exactly the symptom. (The `int` variant works because 8-byte fields sit at 8-aligned offsets, so a full-word read at each offset is correct; the bug is uniquely sub-word.) Note this is in `BC_EXTRACT` itself and is therefore NOT actually iface-dispatch-specific in the bytecode — but the iface-multi-return cell is the only checked-in shape that currently routes a sub-word tuple through extract on the VM, which is why it surfaced there.

### Defect: float-component multi-return packed into INTEGER regs / RAX-RDX byte image, never D0/XMM0 — native↔LLVM ABI divergence (silent wrong value)

**Symptom**: a multi-return tuple with a FLOAT component — `(int, f64)`, `(f64, f64)` — read back as garbage when it crosses the native↔LLVM boundary (e.g. an impl method or multi-return func defined in a non-main, LLVM/clang-compiled package, dispatched from the natively-compiled main module). The natively-compiled callee packs the float field into a GP register / the RAX-RDX byte image, and the native caller collect reads it from there — native↔native self-consistent, but AAPCS64 / SysV-AMD64 (and the LLVM legalization the cross-package callee uses) return a float eightbyte in D0 / XMM0 (or an SSE-classified aggregate eightbyte in an FP reg). The two conventions disagree → the float component reads from the wrong register class. Now reachable through iface dispatch too (post-SEAM, `6c39d460`).

**Root cause (confirmed)**: the native multi-return callee pack and caller collect each have only TWO arms — aggregate vs else-scalar-into-X/RAX-RDX — with no `IsFloatScalarTyp` branch and no HFA/SSE eightbyte classification:
- aa64 callee pack: `aarch64_dispatch.bn` OP_RETURN multi-return loop (lines ~360-385) — scalar arm does `aarch64.Mov(..., argReg(rIdx), ...)` into X0..X7 for EVERY non-aggregate field; only the LONE single-scalar-float early return (`aarch64_dispatch.bn:346`, `if len(ins.Args)==1 && common.IsFloatScalarTyp(...)` → `Fmov_gp_to_fp(D0,...)`) is float-aware.
- aa64 caller collect: `aarch64_call.bn` multi-return loop (lines ~243-262) — scalar arm does `emitScalarStore(a, argReg(rIdx), ...)` reading X0..X7; the float-aware path is ONLY `collectScalarReturn` (`aarch64_call.bn:271-281`, `IsFloatScalarTyp` → `Fmov_fp_to_gp(ftmp, D0)`), used for the single-return case, NOT inside the multi-return loop.
- x64 callee pack: `x64_return.bn` `emitMultiReturnPack` (lines 155-196) — scalar arm does `emitScalarStore(a, src, x64.RSP, base+fldOff, fldSz)` into the integer red-zone image; no float branch. Only the single-scalar-float return (`x64_return.bn:60`, in `emitReturn`) is float-aware.
- x64 caller collect: `collectMultiReturnTuple` (`x64_call.bn:103-112`) stores RAX/RDX raw — it reads the callee's coalesced INTEGER byte image; correct only because the matching callee packed into RAX/RDX, divergent from a SysV callee that would put a float eightbyte in XMM0. (The `IsFloatScalarTyp` branch at `x64_call.bn:157` is in `emitCall`'s ARG emission, not the tuple collect — it handles float *arguments* via XMM, unrelated.)

**Fix shape**: add `IsFloatScalarTyp` handling — and HFA/SSE eightbyte classification — to BOTH the multi-return callee pack and the caller collect on BOTH arches, matching AAPCS64 / SysV-AMD64 + the LLVM legalization. aa64: in the OP_RETURN multi-return loop, a float field of width 4/8 goes to D(nsrn) via `Fmov_gp_to_fp` (track an NSRN counter alongside `rIdx`); a `(f64,f64)` is an HFA → D0,D1. In the caller collect loop, FMOV the corresponding D-reg back to a GP spill slot (mirror `collectScalarReturn`'s float arm, per field). x64: classify each eightbyte INTEGER vs SSE (a `(int,f64)` is INTEGER+SSE → RAX + XMM0; `(f64,f64)` → XMM0+XMM1); the callee `emitMultiReturnPack` must place a float eightbyte's bits into XMM(n) and the caller `collectMultiReturnTuple` must `Movq_xmm_to_gp` it back into the spill image. This is the SysV two-eightbyte classification, not a byte-image coalesce. Confine to per-backend pack/collect; the shared `common.ArgWords` / classifier stays correct as-is.

**Files/functions**: `pkg/binate/native/aarch64/aarch64_dispatch.bn` (OP_RETURN multi-return pack loop ~360-385); `pkg/binate/native/aarch64/aarch64_call.bn` (`emitCall` multi-return collect loop ~243-262, model the float arm on `collectScalarReturn` ~271-281); `pkg/binate/native/x64/x64_return.bn` (`emitMultiReturnPack` scalar arm ~177-186); `pkg/binate/native/x64/x64_call.bn` (`collectMultiReturnTuple` ~103-112 — must SSE-classify, not just store RAX/RDX). ### Defect: native call-conv mis-classifies a `readonly`-wrapped >16-byte aggregate parameter — by-value signature vs by-pointer call site (garbage / SIGSEGV)

**Symptom**: a parameter typed `readonly Big` (24-byte struct) / `readonly [4]int` / `readonly @[]int` is treated as a register-value by the native aggregate classifier but the >16-byte byval/sret convention disagrees → the callee's signature and the caller's pass-by-pointer setup are inconsistent, yielding silent garbage (`func first(b readonly Big) int { return b.a }; first(x)` with `x.a=123` → garbage, expected 123) or SIGSEGV. Controls — plain `Big`/`@[]int`, a below-16B readonly struct, alias-typed slices — all work; only readonly-wrapped >16B aggregates diverge. (The LLVM side — `isByvalParam` in `emit_util.bn`/`gen_func.bn` — is Plan-2; this is the native byval/sret-classifier side.)

**Root cause (confirmed)**: native `common.IsAggregateTyp` (`pkg/binate/native/common/common.bn:346-356`) peels only `t = UnwrapNamed(t)` before the Kind test, and `UnwrapNamed` (`common_scalar.bn:49-53`) loops ONLY on `TYP_NAMED` — it never peels `TYP_READONLY` or `TYP_ALIAS`. So a `readonly Big` param has Kind `TYP_READONLY`, `UnwrapNamed` leaves it as-is, and `IsAggregateTyp` returns FALSE → the param is classified as an in-register scalar instead of a >16-byte byval/sret aggregate. The AAPCS64 / SysV byval threshold (`> 16` / `AggregateInRegMax` / `IndirectLargeAggregates` in `common_callconv.bn`) is keyed on this classification + `SizeOf()` (which DOES peel readonly, via `types.SizeOf`), so the classifier and the size gate disagree exactly the way the LLVM `isByvalParam`/`SizeOf` pair does. Same wrapper-transparency class as Round-1 Defect 1's struct-ptr field-read peel, in the calling-convention layer the Round-1 fixes never reached.

**Fix shape**: peel transparent wrappers (readonly + named + alias, to fixpoint) at the top of `IsAggregateTyp` — and at any sibling native classifier that gates on Kind before the size test (audit `IsFloatScalarTyp` and the aggregate-arg / sret helpers in `common.bn`/`common_call.bn`/`common_callconv.bn` that consume the same Kind). Concretely, either extend `UnwrapNamed` to also peel `TYP_READONLY`/`TYP_ALIAS` (it is the shared backend peel — but check no caller relies on readonly surviving), or add a `peelTransparent`-style helper used by the classifiers. Must mirror the LLVM-side `isByvalParam` peel (Plan-2) so the two backends agree. Confine to the native `common` classifiers; do NOT change `types.SizeOf` (already peels).

**Files/functions**: `pkg/binate/native/common/common.bn` (`IsAggregateTyp` ~346, and confirm the `> 16` byval consumers); `pkg/binate/native/common/common_scalar.bn` (`UnwrapNamed` ~49 — if the peel is widened there) and/or a new peel helper; cross-check `common_callconv.bn` (the AAPCS/SysV byval thresholds). Coordinate with the Plan-2 LLVM `isByvalParam`/`gen_func.bn` peel so both sides land together.

**Test**: per the todo, NO existing test passes a readonly aggregate >16B as an argument. Add WITH the fix — conformance passing a `readonly` >16B struct / array / `@[]T` BY VALUE to a direct call, plus plain / below-threshold / alias controls, on native aa64 + x64 (and LLVM + VM for the Plan-2 companion); xfail until fixed. A native classifier unit test asserting `IsAggregateTyp(readonly Big) == true` pins it below conformance.

**Relation**: sibling of Round-1 Defect 1 (`27c1ee8b` peeled readonly/named/alias in struct-ptr field-read dispatch) — same R1 wrapper-transparency root cause (peeled in predicates but not at the consuming site), here in the native calling-convention classifier rather than the field-read dispatch. The DIRECT-call >16B byval case is the variant the iface/func-value byval work and the field-read peel both missed.

### Defect: native iface-upcast lowerings use `IfaceParentSlotOffset`'s −1 sentinel as a vtable byte offset — silent base-corruption masked by the `byteOff > 0` guard

**Symptom**: `OP_IFACE_UPCAST` computes the target vtable slot via `IfaceParentSlotOffset(src, target)`, which returns −1 when `target` is not a (transitive) PARENT of `src` (and, per the filed amendment, for the SAME canonical interface `(X,X)` — the `any` case is special-cased to 0 but same-interface is not). On the native backends a −1 → `byteOff = -8`, which the `if byteOff > 0` guard silently skips, leaving the vtable pointer at offset 0 (accidentally "correct" only for zero-method interfaces that never deref the vtable). On LLVM the same −1 is written verbatim (`vtable − 8`) → wrong method slot → SIGSEGV. So the SAME IR yields base on native vs base−1 on LLVM — a cross-backend divergence and a latent memory-safety footgun: any future mis-emitted upcast becomes silent wrong-dispatch (native) / crash (LLVM) instead of a loud error. (FILED, user-deferred — the native parts are this plan's.)

**Root cause (confirmed)**: native dispatch lowerings feed `IfaceParentSlotOffset`'s result directly into a byte offset guarded only by `> 0`:
- aa64: `aarch64_dispatch.bn:153-168` — `offsetSlots = ir.IfaceParentSlotOffset(...)`, `var byteOff int = offsetSlots * 8`, then `if byteOff > 0 { aarch64.Add(a, true, vtTmp, vtTmp, aarch64.Imm(byteOff)) }` — a −1 (`byteOff = -8`) falls through the guard, the Add is SKIPPED, the vtable pointer is left unchanged (offset 0), no assert.
- x64: `x64_dispatch.bn:229-244` — identical: `byteOff = offsetSlots * 8`, `if byteOff > 0 { x64.Add(a, RAX, x64.Imm(byteOff)) }` — same silent skip for −1.

The `> 0` guard exists to skip the no-op for a real offset-0 upcast, but it also silently swallows the −1 sentinel — so a genuine sentinel (a non-parent or same-interface upcast that SHOULD be a hard error) is indistinguishable from a legitimate offset-0 upcast.

**Fix shape**: make −1 a HARD error in both native lowerings (not a silent GEP/offset): after computing `offsetSlots`, if it is < 0, emit a compile-time abort / assert (the upcast is ill-formed — a −1 must never reach lowering). Keep the offset-0 fast path (skip the Add for `byteOff == 0`) but distinguish it from the −1 sentinel — guard on `byteOff > 0` for the Add AND assert `offsetSlots >= 0` separately, so a −1 is a loud failure rather than an accidental offset-0. This must land in lockstep with the LLVM lowering (`emit_iface_upcast.bn`, Plan-2) and the `IfaceParentSlotOffset(X,X) → 0` fix in `gen_iface_extends.bn` (Plan-1) so all three lowerings treat −1 as an assert and same-interface as a true 0. Plan 3 owns ONLY the two native dispatch sites; the `IfaceParentSlotOffset` semantics change and the LLVM lowering are flagged as Plan-1/Plan-2 dependencies.

**Files/functions**: `pkg/binate/native/aarch64/aarch64_dispatch.bn` (`OP_IFACE_UPCAST` arm, the `offsetSlots`/`byteOff`/`byteOff > 0` block ~153-168); `pkg/binate/native/x64/x64_dispatch.bn` (`OP_IFACE_UPCAST` arm ~229-244). Read-only dependency on `pkg/binate/ir/gen_iface_extends.bn` `IfaceParentSlotOffset` (Plan-1, the `(X,X) → 0` fix) and `pkg/binate/codegen/emit_iface_upcast.bn` (Plan-2, the LLVM −1-as-assert).

**Test**: currently latent — no remaining known emitter of a −1-offset upcast after the alias-canonicalization fix (`a869e8e7`) stopped the alias path. A direct test needs an emitter; pin instead via a native unit test that constructs an `OP_IFACE_UPCAST` with a non-parent (src,target) pair and asserts the lowering ABORTS (does not silently skip), plus — once `IfaceParentSlotOffset(X,X)` returns 0 — a same-interface managed↔raw decay cell confirming offset 0 (not −1) across all three backends. The unit-test comment at `gen_iface_extends_test.bn:72-75` claiming the caller "leaves offset at 0" is aspirational (no caller clamps) and should be corrected to match the asserting behavior.

**Relation**: native half of the FILED CRITICAL "Iface upcast lowerings use `IfaceParentSlotOffset`'s −1 sentinel" (the Plan-cr2-1 Defect 8 follow-up). Defect 8 (`a869e8e7`) stopped the ALIAS re-export path from EMITTING the −1 upcast, but the lowerings still trust the offset blindly — this is the sibling that hardens the consuming side so any other non-parent / same-interface upcast becomes a loud error instead of native silent-wrong-dispatch / LLVM crash. Sub-part (a) of that entry; sub-part (b) (`IfaceParentSlotOffset` itself) is Plan-1.
