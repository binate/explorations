# Code-Red-2 ‚Äî Plan 3: Backends (native aa64/x64/arm32 + bytecode VM)

> One of three disjoint Code-Red-2 fix plans (mirrors the P2 partition in `plan-cr-p2-1-frontend.md` / `plan-cr-p2-3-abi.md`).
> Source-confirmed against `temp-binate-4/pkg/binate/native/{aarch64,x64,common}` and `pkg/binate/vm` (READ-ONLY; line numbers are what the current tree shows). Defects of record live in `claude-todo.md`; this is the execution view.

## Theme

Every defect here is one unfilled cell of the FEATURE √ó MODIFIER √ó POSITION product, and all six fail the two Code-Red-2 invariants directly. PATH PARITY: the sub-word-narrow re-reduction, the eightbyte-coalescing multi-return pack/collect, and the small-aggregate funcval-return convention each got fixed in ONE lowering path (the binary-op path, the *direct*-call path) but not in the parallel paths (the unary-op path, the iface-method and func-value collect paths, the small-aggregate shim) ‚Äî so a value of the exact triggering shape silently breaks where its sibling passes. WRAPPER TRANSPARENCY shows up in the VM's BC_EXTRACT, which reads a full host word regardless of the field's sub-word type, and in the global-materialization parity work that must peel the same `TYP_READONLY` / named wrappers codegen's global emitter now peels. The fixes are confined to the backend packing/narrowing/materialization sites; none touches `pkg/binate/ir`, `pkg/binate/types`, or `pkg/binate/codegen`.

## Defects

### Defect: native `emitUnop` (OP_NEG / OP_BITNOT) omits the sub-word re-narrow that `emitBinop` performs (PATH-PARITY gap)

**Symptom**: a sub-word unary result (`~x` / `-x` for `uint8/16/32`, `int8/16/32`) leaves dirty upper bits in the host register on aa64 and x64. Latent for most consumers because the next sub-word op / compare / cast re-narrows; exposed when the un-narrowed value is consumed directly by a widening read, an unsigned compare, a shift count, or a store of the full word.

**Root cause (confirmed)**: aa64 `aarch64_ops.bn` ‚Äî `emitBinop` ends with `emitSubWordNarrow(a, rd, ins.Typ)` at line 123 before `return true`. `emitUnop` (lines 187-214) emits `Neg` (line 201) / `Mvn` (line 205) and immediately `return true` with NO `emitSubWordNarrow` call. x64 `x64_ops.bn` ‚Äî identical asymmetry: `emitBinop` calls `emitSubWordNarrow(a, rd, ins.Typ)` at line 156, and `emitDivOrRem` at line 217, but `emitUnop` (lines 340-372) emits `Neg` (line 359) / `Not` (line 363) and returns with no narrow. The shared classifier `common.SubWordNarrow` and the per-backend `emitSubWordNarrow` (aa64 lines 20-34, x64 lines 29-45) already exist and are no-ops for host-word / non-integer types, so the call is safe to add unconditionally. (OP_NOT ‚Äî the bool path, aa64 line 209 / x64 line 367 ‚Äî needs no narrow: it XORs bit 0 of a 1-bit value.)

**Fix shape**: append `emitSubWordNarrow(a, rd, ins.Typ)` immediately before the `OP_NEG` and `OP_BITNOT` `return true`s in both `emitUnop`s ‚Äî mirroring `emitBinop` exactly. Cleanest: narrow once just before the final `return true` for the NEG/BITNOT arms (skip the NOT arm). `~` on a sub-word is the most exposed: `~x` over a `uint8` should yield an 8-bit complement; without the uxtb the high bits carry the host-width complement.

**Files/functions**: `pkg/binate/native/aarch64/aarch64_ops.bn` (`emitUnop`, the OP_NEG arm ~line 200-203 and OP_BITNOT arm ~line 204-207); `pkg/binate/native/x64/x64_ops.bn` (`emitUnop`, OP_NEG arm ~line 358-361 and OP_BITNOT arm ~line 362-365). Reuses the existing `emitSubWordNarrow` in each file; touches NO shared/common file.

**Test coverage**: the matrix cell `conformance/matrix/scalar-diff` covers sub-word arith on the natives but, per `claude-todo.md` (`aa64-subword`, lines ~904-924), the unary `~`/`-` directly-consumed cell is NOT pinned ‚Äî the `bitwise/not` cells exercise `~` but through LLVM/IR-gen typing (the `bitnot-result-type` fix), not the native re-narrow. GAP to fill: a probe that consumes a sub-word `~`/`-` result WITHOUT re-narrowing ‚Äî e.g. `var y uint16 = ~x; print(cast(uint64, y))` or `if (~x) > k { ‚Ä¶ }` where the dirty high bits change the unsigned comparison ‚Äî added as a `scalar-diff`-style cell or a native unit test in `aarch64_ops_test.bn` / `x64_ops_test.bn` asserting the uxt/sxt (or movzx/movsx) is emitted after NEG/BITNOT. The probe MUST avoid an intervening cast/compare that would mask the omission.

### Defect: aa64 mis-packs a sub-word / non-8-multiple struct returned through a FUNCTION-VALUE call (`funcval-return/five-u8`)

**Symptom**: `conformance/matrix/abi/funcval-return/five-u8` ‚Äî a `@func() S` returning a 5√óuint8 struct `{1,2,3,4,5}` prints `16,146,211,‚Ä¶` on native aa64 (and x64). The interface-dispatch sibling `iface-return/five-u8` and the DIRECT struct-return both pass; only the function-value-call result path mis-packs (`claude-todo.md` top entry, lines 33-36, marked CRITICAL/CONFIRMED).

**Root cause (confirmed)**: a small (‚â§ 8-byte) aggregate funcval return hits a CONVENTION MISMATCH between the caller's collect and the per-function shim. In `aarch64_call_indirect.bn`, `emitCallFuncValue` sets `aggregateRet` true for any non-multi-return `IsAggregateTyp(ins.Typ)` (lines 241-243) ‚Äî a 5-byte struct qualifies ‚Äî and then uses the **retbuf-in-X0** convention: it passes `X0 = SP + retbufOff` (lines 356-361) and after the call just produces the alloc-region pointer (lines 394-399), trusting the shim to have written the result through the retbuf. But the shim (`aarch64_funcvalue.bn` `emitFuncValueShims`) only builds a retbuf-aware body when `usePack` (`retSz > 8 && retSz <= 16`, line 213) or `useSret` (`retSz > 16`, line 212); a `retSz <= 8` aggregate falls into the **scalar/void shim** (lines 267-276), which ignores X0/retbuf, tail-branches, and lets the underlying return the packed struct in X0. The caller then reads an UNWRITTEN retbuf ‚Üí garbage. (The `retWords = retSz/8; if retSz % 8 != 0 { retWords++ }` at lines 245-246 is a correct ceil and not the bug.) x64 is the same shape: `x64_call_indirect.bn` `aggregateRet` (lines 225-227) + retbuf-in-RDI (lines 284-292), with the x64 shim's `usePack` gated `retSz > 8` (line 183) so a ‚â§ 8-byte aggregate takes the scalar shim (lines 228-234).

**Fix shape**: close the missing convention arm for 1..8-byte aggregate funcval returns. Two equivalent routes; pick one and apply to BOTH backends symmetrically: (a) extend the shim's pack path to fire for `retSz >= 1 && retSz <= 16` (lower the `usePack` floor and have the ‚â§ 8-byte case store just RAX/X0 through retbuf) so the caller's existing retbuf convention is honored; OR (b) make `emitCallFuncValue` treat a ‚â§ 8-byte aggregate return as a register return (drop `aggregateRet` for `SizeOf <= 8`, collect RAX/X0 into the data region like the small-aggregate DIRECT-call path in `x64_call.bn:256-273` / `aarch64_call.bn:206-221`). Route (a) keeps the funcval ABI uniform (retbuf always meaningful for aggregates); route (b) keeps the shim minimal. Whichever: caller and shim MUST agree, and the choice must match how `emitFuncValueShims`'s `shimReturnSize` thresholds are read on the same backend. Confirm against the direct small-aggregate return, which already works.

**Files/functions**: `pkg/binate/native/aarch64/aarch64_funcvalue.bn` (`emitFuncValueShims`, the `usePack`/`useSret`/scalar split ~lines 211-276; `shimReturnSize` ~lines 133-138); `pkg/binate/native/aarch64/aarch64_call_indirect.bn` (`emitCallFuncValue`, `aggregateRet` ~241-245, retbuf setup ~356-364, collect ~394-399); `pkg/binate/native/x64/x64_funcvalue.bn` (`emitFuncValueShims` ~181-234; `shimReturnSize_x64` ~120-125); `pkg/binate/native/x64/x64_call_indirect.bn` (`emitCallFuncValue`, `aggregateRet` ~225-229, retbuf ~284-292, collect ~336-342).

**Test coverage**: pinned by `conformance/matrix/abi/funcval-return/five-u8` (xfail native aa64/x64, likely arm32) ‚Äî flips to XPASS on fix. GAP: a 1-8-byte funcval return that is NOT sub-word-packed (e.g. `three-u16` = 6B, or a `{int32}` 4B) to prove the fix covers the whole ‚â§ 8 aggregate-return-via-funcval region, not just the five-u8 cell; plus a 9..16-byte funcval-return cross-check that the existing `usePack` path still passes (regression guard for route (a)).

### Defect: x64 sub-word multi-return mis-collected through the INDIRECT call shapes (iface-method + func-value) ‚Äî PATH-PARITY gap vs the fixed direct path

**Symptom**: a sub-word (`uint16`) multi-return at arity ‚â• 3 returned through an iface-method dispatch or a func-value call mis-packs the 3rd+ component on x64 native (silent wrong values). The DIRECT-call sub-word multi-return was fixed (binate `b5616b32`) but the indirect collect sites were not updated to match.

**Root cause (confirmed)**: the callee now packs the tuple as a COALESCED SysV byte image in RAX (+RDX) ‚Äî `x64_return.bn` `emitMultiReturnPack` (lines 155-196) materializes the tuple in the red zone at each field's `FieldOffset`, coalescing multiple sub-word fields into one eightbyte, then loads RAX/RDX. The DIRECT collect in `x64_call.bn` (lines 230-247) matches it: it stores RAX/RDX RAW into the spill slot and lets each OP_EXTRACT read its component at the right offset. But the INDIRECT collects still use the OLD per-field-register scheme: `x64_iface.bn` `emitCallIfaceMethod` (lines 191-208) does `var n = len(ins.Typ.Fields); if n > 2 { n = 2 }` then `srcReg = RAX` for field 0, `RDX` for field 1, `emitScalarStore` per field ‚Äî dropping field 2+ and reading field 1 from RDX (which for a coalesced `(u16,u16,u16)` holds nothing). `x64_call_indirect.bn` `emitCallFuncValue` has the identical stale block (lines 345-362, `if n > 2 { n = 2 }`, field0‚ÜíRAX/field1‚ÜíRDX). aa64 is NOT affected: its iface/funcval collects (`aarch64_iface.bn` lines 208-227, `aarch64_call_indirect.bn` lines 409-428) store `argReg(rIdx)` per word with sized `emitScalarStore`, consistent with the aa64 callee returning one register per tuple word.

**Fix shape**: make the x64 iface-method and func-value multi-return collects use the SAME raw-RAX/RDX-into-spill scheme as the fixed direct path (`x64_call.bn:234-246`): drop the `if n > 2 { n = 2 }` cap and the per-field RAX/RDX split; for a `‚â§ 16`-byte tuple store RAX ‚Üí `[spill+0]` and (if `SizeOf > 8`) RDX ‚Üí `[spill+8]` raw, so each downstream OP_EXTRACT reads its coalesced component. (Big multi-return `> 16B` already short-circuits via `bigMultiRet` on all three.)

**Files/functions**: `pkg/binate/native/x64/x64_iface.bn` (`emitCallIfaceMethod` multi-return collect ~lines 191-208); `pkg/binate/native/x64/x64_call_indirect.bn` (`emitCallFuncValue` multi-return collect ~lines 345-362); optionally a new shared helper beside `x64_call.bn`'s `emitCall` collect (~lines 230-246) that all three call. Touches NO common/shared-ABI file (the byte-image scheme is x64-local; `common.ArgWords` and the classifier are already correct).

**Test coverage**: the direct cells `conformance/matrix/abi/multi-return/u16/{3,4,5}` are green post-`b5616b32`, but the INDIRECT shapes are the gap: `conformance/matrix/abi/iface-multi-return/u16/{3,4,5}` and `funcval-multi-return/u16/{3,4,5}` (the latter currently blocked at type-check by the separate funcval-multi-return front-end defect, `claude-todo.md` lines 19-25 ‚Äî coordinate so its cells exercise x64 codegen once the front-end accepts them). ### Defect: VM mis-unpacks a sub-word (uint16) multi-return through interface dispatch ‚Äî BC_EXTRACT reads a full host word (WRAPPER/width-TRANSPARENCY gap)

**Symptom**: `conformance/matrix/abi/iface-multi-return/u16/2` ‚Äî a dispatched `mr() (uint16, uint16)` returning `(100, 200)` prints `13107300, 1` on the bytecode VM instead of `100, 200`. `13107300 = (200<<16)|100` ‚Äî the first extracted component reads BOTH u16s out of word 0. The `int` variant is correct on the VM (`claude-todo.md` lines 27-30, CONFIRMED).

**Root cause (confirmed)**: the multi-return tuple is packed as a correct sub-word byte image, but `BC_EXTRACT` reads a full 8-byte word with no sub-word sizing. Pack side (`vm_exec_return.bn` `packMultiReturn`, lines 24-66): for a non-pair non-multiword scalar field it writes `var p *int = bit_cast(*int, dstBase + off); p[0] = regs[...]` (lines 52-55) ‚Äî an 8-byte store at the field's `ResultOffsets[i]` (computed with natural alignment in `lower_func.bn:64-78`, so `(u16,u16)` ‚Üí offsets 0 and 2). Reading the image back: lowering (`lower_instr.bn` OP_EXTRACT arm, lines 358-391) sets `bc.Imm = srcTyp.FieldOffset(instr.Index)` (line 369) and only marks pointer mode (`bc.Aux = 1`) for `isMultiWordField || isVMAddressAggregate` (lines 380-383) ‚Äî a sub-word scalar field stays value-mode with `Aux == 0`. The exec (`vm_exec_helpers.bn` `BC_EXTRACT`, lines 430-447): the `Aux == 0` arm (lines 442-445) does `var p *int = bit_cast(*int, regs[Src1] + Imm); regs[Dst] = p[0]` ‚Äî a FULL `*int` (8-byte) load at the byte offset, with no width mask. Extracting field 0 of the `(u16,u16)` image reads bytes[0:8] = `100 | (200<<16)` = 13107300 ‚Äî exactly the symptom. (The `int` variant works because 8-byte fields sit at 8-aligned offsets, so a full-word read at each offset is correct; the bug is uniquely sub-word.) Note this is in `BC_EXTRACT` itself and is therefore NOT actually iface-dispatch-specific in the bytecode ‚Äî but the iface-multi-return cell is the only checked-in shape that currently routes a sub-word tuple through extract on the VM, which is why it surfaced there.

**Fix shape**: make value-mode `BC_EXTRACT` perform a SIZED sub-word load + correct extension, mirroring `BC_LOAD8`'s sized read (`vm_exec_helpers.bn:388-396`, which already does a byte-width load with optional sign-extend via its `Aux == 1` flag). Carry the field's byte size (and signedness) from lowering: in `lower_instr.bn`'s OP_EXTRACT arm, when the field is a sub-word scalar (`ft.SizeOf() < 8`), record the size/signedness (e.g. via a new size field or by reusing the `Aux`/a spare operand) instead of plain value-mode; in `vm_exec_helpers.bn`'s `BC_EXTRACT` value arm, read exactly that many bytes (`uint8`-indexed assemble, like `BC_LOAD8`) and zero/sign-extend per the field type. Keep the pointer-mode (`Aux == 1`) and legacy word-indexed (`Aux == 2`) arms unchanged; full-word fields keep the current 8-byte fast path. WRAPPER TRANSPARENCY: the field type must be unwrapped (`vmUnwrapNamed` / `ResolveAlias`) before reading width/signedness so a named sub-word component (`type U16 uint16`) is sized correctly.

### Defect: VM global-materialization parity with the codegen iface/func/managed-slice/named global changes across `-int` modes

**Symptom**: the LLVM global emitter was fixed (binate `5dddef7d` / landed `91ef4fc4`) to emit `zeroinitializer` for the four 16-byte address-aggregate kinds AND to peel `TYP_READONLY` before the static-zero dispatch (`claude-todo.md` lines 205-216).
