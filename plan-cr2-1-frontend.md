# Code-Red P2 (CR-2) — Plan 1: Front-end IR-gen + type-checker (wrapper-transparency + path-parity defects)

> One of three disjoint Code-Red-2 fix plans. This plan owns `pkg/binate/ir` and `pkg/binate/types` ONLY; it touches no `pkg/binate/codegen`, `pkg/binate/native`, or `pkg/binate/vm` file.
> Every defect below was source-confirmed against the live tree under `/Users/vtl/binate/temp-binate-4/pkg` (line numbers cited are what I found; the earlier todo/plan citations had drifted). Defect-of-record entries live in `explorations/claude-todo.md`.

## Theme

Eight front-end defects, all instances of the two CR-2 meta-invariants. (a) WRAPPER TRANSPARENCY — a `Kind`-dispatch that forgets to peel `TYP_READONLY` / `TYP_NAMED` (and `TYP_ALIAS`) before testing the wrapped kind, so a `readonly @Box` / named-distinct value sails past every arm and silently materializes literal-0 (defect 1) or hard-rejects dispatch (defect 2). (b) PATH PARITY — a feature lowered in N parallel paths fixed in only some of them: interface-method RESULTS never got the params-flat generalization the PARAMS side did (defect 3); the multi-return DESTRUCTURE checks accept `TYP_FUNC` but not the `TYP_FUNC_VALUE` siblings (defect 4); the nested-composite alloca-load guard fires for struct elements but not array elements (defect 6); the expression-form shift routes through the overshift guard but the compound-assign form does not (defect 7). Defect 5 is a missing checker guard (aggregate `==`/`!=` accepted → invalid `icmp`). Defect 8 (transitive re-exported interface) is the one speculative item — flagged below as unconfirmed-reproducer. The unifying repair is the same in every case: route the dropped shape through the machinery the working sibling already uses, after peeling wrappers at the choke point.

## Defects

### Defect 1: `readonly` (and named-wrapped) managed/raw struct-pointer field read silently returns literal-0

**Symptom**: `var p @Box = make(Box); p.v = 7; var rp readonly @Box = p; println(rp.v)` prints `0`, not `7` (direct `p.v` prints `7`). Silent data corruption. Same shape for any `TYP_NAMED`-wrapped pointer-to-struct (`type Handle @Box; var h Handle; h.v`). Confirmed on LLVM/compiled in `claude-todo.md` (the "`readonly` / `const` type modifier is broken for managed values" CRITICAL, Symptom 1).

**Root cause (confirmed)**: `pkg/binate/ir/gen_selector.bn`. `genSelector` (the field-read path) and `genSelectorPtr` (the lvalue path) dispatch on the *raw* variable type via three Kind-tests that do NOT peel wrappers:
- `gen_selector.bn:20` — `varTyp.Kind == types.TYP_STRUCT` (value struct arm).
- `gen_selector.bn:30,46` — `isManagedPtrToStruct(varTyp)` / `isRawPtrToStruct(varTyp)` (pointer arms), and the parallel sites at `:75,88,106,118,149,162,192,227,238` plus the lvalue mirrors in `genSelectorPtr` at `:313,322,334,352,361,373,389,399,416,425,437`.

The predicates themselves are non-transparent: `gen_refcount_pred.bn:119-123` `isManagedPtrToStruct` tests `t.Kind == types.TYP_MANAGED_PTR && t.Elem.Kind == types.TYP_STRUCT` with no peel; `:126-130` `isRawPtrToStruct` likewise; `:15-18` `isManagedPtrType` likewise. For a `readonly @Box`, `varTyp.Kind == TYP_READONLY`, so all three fail, every arm of `genSelector` is skipped, and control reaches the fallback `gen_selector.bn:297` `return b.EmitConstInt(0, types.TypInt())` — literal 0, type-independent. (Verified the parallel `llvmType` choke point peels both `TYP_READONLY` and `TYP_NAMED`/`Underlying` at `pkg/binate/codegen/emit_types.bn:59,61-62`, so codegen would have emitted the right LLVM type had IR-gen produced the GEP — the bug is purely that IR-gen never produces it.)

**Fix shape**: Make the predicates wrapper-transparent at the choke point — peel `TYP_READONLY` and `TYP_NAMED` (mirroring `llvmType`) inside `isManagedPtrToStruct`, `isRawPtrToStruct`, and `isManagedPtrType`. The peel helpers already exist: `peelReadonly` (`gen_refcount_pred.bn:69-72`) and `peelNamed` (`gen.bn:315-318`); compose them (a small `peelNamedAndReadonly` loop, or call both to a fixpoint) at the top of each predicate, then test `.Kind`/`.Elem` on the peeled type. For `genSelector`/`genSelectorPtr`'s direct `varTyp.Kind == TYP_STRUCT` tests (`:20,65,130,177,203,249,313,352,416`) and every `.Elem` deref that follows a predicate, read from a peeled `varTyp`/`elemTyp`/`innerTyp` so the struct index + field type resolve against the underlying struct. Because the predicates and the `.Elem` reads are now peeled, the field GEP + load fire for the readonly/named value exactly as for the bare pointer. Do NOT alter the field offset logic — readonly/named are layout-transparent (verified `MakeReadonlyType` keeps size/align in `types_const.bn:7-20`), so the same `lookupFieldIndex`/`lookupFieldType`/`EmitGetFieldPtr` is correct once the wrapper is off.

**Files/functions**: `pkg/binate/ir/gen_refcount_pred.bn` (`isManagedPtrToStruct` ~:119, `isRawPtrToStruct` ~:126, `isManagedPtrType` ~:15 — add the peel); `pkg/binate/ir/gen_selector.bn` (`genSelector` direct `TYP_STRUCT` tests + `.Elem` reads ~:20,30,46,65,75,88; `genSelectorPtr` mirrors ~:313,322,334,352,361,373). Reuse `peelReadonly`/`peelNamed` (no new file).

**Test coverage**: GAP — no conformance cell exists for readonly/named field read (`claude-todo.md` lists "Tests (TODO with fix)"). Matrix cell to fill: readonly-managed field read (value-correct, all modes) + a named-distinct-over-`@struct` companion. Add a `gen_selector` unit test asserting the readonly/named receiver produces an `OP_GET_FIELD_PTR` + `OP_LOAD` (not `OP_CONST_INT 0`). This is Symptom 1 of the readonly CRITICAL; Symptom 2 is Defect 2 below.

### Defect 2: `readonly @Iface` (and `readonly @struct` with methods) cannot be dispatched — hard compile error

**Symptom**: any method call on a `readonly` managed value fails at type-check: `readonly @Iface` → `cannot access field on this type`; `readonly @Box` → `cannot call method Get: receiver type readonly @Box not assignable to @Box`. Since all interface use IS dispatch, `readonly @Iface` is entirely unusable. `claude-todo.md` readonly CRITICAL, Symptom 2.

**Root cause (confirmed)**: `pkg/binate/types/check_method.bn:37` `tryMethodCall` does `var resolved @Type = recvType.ResolveAlias()`, then tests `resolved.Kind == TYP_INTERFACE_VALUE / TYP_INTERFACE_VALUE_MANAGED` at `:38-42` to route to `tryInterfaceMethodCall`. But `ResolveAlias` (`types_query.bn:120-125`) follows ONLY `TYP_ALIAS` — it does not peel `TYP_READONLY`. So a `readonly @Iface` keeps `resolved.Kind == TYP_READONLY`, never enters the iface-dispatch arm, falls through to `recvType.ReceiverBaseNamed()` (`:72`) which can't see an interface base, and errors. The pointer-to-iv smoothing at `:50-59` (`resolved.Elem.ResolveAlias()`) has the same gap one layer down. (For `readonly @struct`-with-methods: `receiverShape`/`receiverAssignable` DO handle readonly via `resolveAliasAndConst` and a `srcConst && !dstConst` check at `:230-234,310-312`, so the const-method-receiver rejection is a *separate, intended* policy decision — see "Open spec note" — not the same Kind-dispatch miss. This plan fixes the iface-dispatch miss; the const-receiver-method-policy is a language-design call to flag to the user, not silently change.)

**Fix shape**: The helper the plan names already exists — `resolveAliasAndConst` (`types_const.bn:44-55`) peels both `TYP_ALIAS` and `TYP_READONLY` to a fixpoint. Do NOT add a new helper. At `check_method.bn:37`, replace `recvType.ResolveAlias()` with `resolveAliasAndConst(recvType)` (package-level function call, not a method) so a `readonly @Iface` reaches the `TYP_INTERFACE_VALUE(_MANAGED)` arm; mirror at the pointer-to-iv inner peel `:54` (`resolveAliasAndConst(resolved.Elem)`). This strips `TYP_READONLY` before the interface-value kind test, exactly the WRAPPER-TRANSPARENCY invariant. NOTE for the user: the `readonly @struct`-method-receiver rejection (Symptom 2's second half) is a deliberate const-correctness question (does a `readonly` receiver implicitly permit non-mutating method calls, à la C++ `const` methods / Rust `&self`?) — `claude-todo.md` flags it as "partly a language-design call." This plan does the iface-dispatch peel (a clear bug — `readonly @Iface` should dispatch) and surfaces the const-receiver-method policy for a user decision rather than unilaterally adding a readonly-receiver assignability path.

**Files/functions**: `pkg/binate/types/check_method.bn` (`tryMethodCall` ~:37 and the pointer-to-iv inner peel ~:54). Reuse `resolveAliasAndConst` from `types_const.bn`.

**Test coverage**: GAP — no readonly-iface dispatch cell. Add a conformance cell (`readonly @Iface` method call, value-correct, all modes) and a `check_method` unit test asserting `readonly @Iface` type-checks the dispatch. Pair with the Defect-1 readonly cells.

### Defect 3: interface-method dispatch drops the result type for any method with ≠1 results → `extractvalue void` (LLVM) / silent wrong-code (native)

**Symptom**: `var i @I = x; a, b = i.M()` where `M() (int, int)` emits `extractvalue void %vN, 0` → clang `error: void type only allowed for function results` (LLVM); native aa64/x64 SILENTLY miscompile (duplicate/garbage values). The idiomatic `(T, @Error)` interface method cannot be dispatched at all. `claude-todo.md` top CRITICAL; blocks `pkg/std/io` (Read/Write all return `(int, @errors.Error)`).

**Root cause (confirmed)**: `pkg/binate/ir/gen_iface_registry.bn`. `ModuleInterface.MethodResults @[]@types.Type` (`:37`) stores exactly ONE result type per method; population at `:176-181` is `var resultTyp = nil; if len(m.Results) == 1 { resultTyp = resolveTypeExpr(m.Results[0]) }` — a method with 0 or ≥2 results stores `nil`. `findInterfaceMethod` → `findInterfaceMethodFromBase` returns that single `rt` (`gen_iface.bn:178-181,186`); `genInterfaceMethodCall` (`gen_iface.bn:83-90,106`) takes `resultTyp`, coerces `nil`→`TypVoid()` at `:90`, and emits `EmitCallIfaceMethod(iv, args, idx, resultTyp)` with a void result. The multi-assign's downstream `extractvalue` against a void result is invalid. The PARAMS side was already generalized to N-per-method via `MethodParamsFlat @[]@types.Type` + `MethodParamCounts @[]int` (`:49-50`, populated `:185-190`, consumed in `findInterfaceMethodFromBase:170-189`); RESULTS never got the same flattening. The same single-result limitation is duplicated at the generic-instantiation site `gen_generic.bn:358-363`.

**Fix shape** — mirror the params-flat treatment (this DEFINES the CR-2 seam field). Add `MethodResultsFlat @[]@types.Type` + `MethodResultCounts @[]int` to `ModuleInterface` (`gen_iface_registry.bn:28-67`), exactly parallel to `MethodParamsFlat`/`MethodParamCounts`. Populate at BOTH decl sites in the existing method loop: `gen_iface_registry.bn:174-191` (flatten every `m.Results[k]` via `resolveTypeExpr`, append `len(m.Results)` to counts) AND `gen_generic.bn:356-371` (same, under the type-param substitution context). Change `findInterfaceMethodFromBase` to return the result LIST (slice `MethodResultsFlat[rOff..rOff+rCount]`, walking `rOff` in lockstep with `j` like `pOffset` does for params at `:171,188`); propagate the list up through `findInterfaceMethod` (`gen_iface.bn:128-139`). In `genInterfaceMethodCall` (`gen_iface.bn:83-107`), build the dispatch result type from the list: 0 results → `TypVoid()` (confirm nil-vs-void is what the void thunk expects — `gen_iv_thunk.bn:209-217` uses `TypVoid` for the no-result case); 1 result → the single type; ≥2 → `makeMultiReturnStructType(results)` (the exact tuple type the direct multi-return call builds at `gen_call.bn:228-235`, `gen_func.bn:44`), then `EmitCallIfaceMethod(iv, args, idx, mrt)` so `extractvalue` lands against a real struct type. Keep `MethodResults` during the transition or retire it once all readers move to `MethodResultsFlat` (params kept only the flat form — follow that precedent). **SEAM**: `MethodResultsFlat`/`MethodResultCounts` are DEFINED here and CONSUMED read-only by Plans 2/3 (native/codegen tuple layout) — do not let Plans 2/3 populate them.

**Files/functions**: `pkg/binate/ir/gen_iface_registry.bn` (`ModuleInterface` struct ~:28-67; population loop ~:174-191); `pkg/binate/ir/gen_generic.bn` (generic-instantiation population ~:356-371); `pkg/binate/ir/gen_iface.bn` (`findInterfaceMethodFromBase` ~:146-191, `findInterfaceMethod` ~:128-139, `genInterfaceMethodCall` ~:83-107). ### Defect 4: func-value multi-return destructure rejected at type-check (`assignment count mismatch`)

**Symptom**: `var fv @func() (int, int) = mr; a, b := fv()` (or `a, b = fv()`, or `return fv()`) → `assignment count mismatch` / `wrong number of return values`. Multi-result *free-function* calls work; the defect is `{function-value call} × {multi-value result}`. Front-end rejection — fails on every mode. `claude-todo.md` MAJOR.

**Root cause (confirmed)**: `pkg/binate/types`. `checkCallExpr` (`check_expr.bn:446-453`) returns the whole callee type for a multi-result call so the assignment can expand it — and for a func-value callee that whole type is `TYP_FUNC_VALUE` / `TYP_MANAGED_FUNC_VALUE` (a func-value's `checkExpr` resolves to one of those kinds), NOT `TYP_FUNC`. But the three expansion sites guard on `rhsType.Kind == TYP_FUNC` exclusively: `check_stmt.bn:156` (`checkAssignStmt` multi-assign), `check_stmt.bn:243` (`checkShortVarDecl` multi-short-var), `check_stmt.bn:287` (`checkReturnStmt` single-expr-multi-return expansion). A func-value call's result never matches → falls to "assignment count mismatch" (`:171`, `:251`) / "wrong number of return values" (`:298`). Verified the func-value kinds carry `.Results` identically (`resolveTypeExpr` builds them with results at `gen_util.bn:397-405`; `funcSignaturesMatch` reads `fv.Results` at `types_query.bn:251,257`).

**Fix shape**: PATH PARITY — at all three guard sites accept the func-value kinds alongside `TYP_FUNC`. Change each `rhsType.Kind == TYP_FUNC` (and the `checkReturnStmt` `rhsType.Kind == TYP_FUNC`) to also admit `TYP_FUNC_VALUE` / `TYP_MANAGED_FUNC_VALUE`; the existing `len(rhsType.Results) == len(s.Exprs)` count check and the per-result `AssignableTo` loop are kind-agnostic and work unchanged. A small shared predicate (e.g. `hasExpandableResults(t)` returning true for the three kinds) keeps the three sites from re-diverging. After the front-end accepts it, the downstream IR-gen/backends for a multi-return func-value call were previously unreachable (the checker blocked them) — verify they lower correctly as part of this fix; the matrix cells (below) flip from front-end-fail to exercising codegen.

**Files/functions**: `pkg/binate/types/check_stmt.bn` (`checkAssignStmt` ~:156, `checkShortVarDecl` ~:243, `checkReturnStmt` ~:287). Optionally add the shared `hasExpandableResults` predicate (in `check_stmt.bn` or `types_query.bn`).

**Test coverage**: matrix cells EXIST — `conformance/matrix/abi/funcval-multi-return/{int,u16}/{2,3,4,5}`, xfailed on ALL modes (front-end rejection). Unxfail the modes where the lowered call is correct after the fix; keep xfails only on modes with a genuine downstream backend gap (e.g. the native `five-u8`/sub-word funcval-return packing CRITICAL is a separate Plan-3 item; confirm per-mode by running rather than blanket-unxfailing). Add a `check_stmt` unit test asserting `a, b := fv()` type-checks.

### Defect 5: `==` / `!=` on an aggregate operand is accepted by the checker, then emits invalid `icmp`

**Symptom**: `a == b` / `a != b` where operands are `*[]int`, `@[]int`, `*func()int`, `@func()int`, `@Iface`/`@errors.Error`, or a struct `Pt` type-checks, then codegen emits a scalar `icmp` on a >1-word value → clang `error: icmp requires integer operands`; whole package fails. `claude-todo.md` CONFIRMED; the *semantics* are an open spec gap, the user's direction (2026-06-07) is DISALLOW for interface values, slices, and func values.

**Root cause (confirmed)**: `pkg/binate/types/check_expr.bn:262-269`. `checkBinaryOp`'s comparison branch (covers `==`/`!=`/`<`/`>`/`<=`/`>=`) does only `if !lt.AssignableTo(rt) && !rt.AssignableTo(lt) { errTypeMismatch }` then `return TypUntypedBool()` — there is NO scalar/pointer operand guard. So two assignable aggregate operands pass, and codegen's equality lowering (which assumes a single-word integer/pointer and emits `icmp`) produces invalid IR.

**Fix shape**: Add a checker reject for `==`/`!=` on aggregate operands (the user-chosen disposition; this plan implements the agreed DISALLOW, it does not invent new equality semantics). In the `==`/`!=` sub-branch of `check_expr.bn:262-269`, after the type-compatibility check, reject when either operand is non-scalar-and-non-pointer: i.e. NOT (`IsScalar()` or `IsPointer()` or `TYP_NIL`). `IsScalar()` (`types_query.bn:207-209`, covers int/float/bool through named/readonly) and `IsPointer()` (`:212-218`, raw + managed pointer) already exist; `nil` comparisons against a pointer stay allowed. This rejects slices, managed-slices, func-values, interface-values, and structs with a clear diagnostic (e.g. "cannot compare values of aggregate type T; use present() for interface/slice emptiness"). Scope per the user decision: DISALLOW iface/slice/func-value definitively; STRUCT equality is still an open spec question (fieldwise vs disallow) — flag to the user whether to reject structs now or defer the struct arm. Either way the current invalid-IR path must end (reject or correct-lowering, never `icmp` on an aggregate). The ordering relational ops (`<`/`>`/`<=`/`>=`) on aggregates should also be rejected here (they're already nonsensical); confirm no test relies on them.

**Files/functions**: `pkg/binate/types/check_expr.bn` (`checkBinaryOp` comparison arm ~:262-269). Reuse `IsScalar`/`IsPointer` from `types_query.bn`.

**Test coverage**: GAP — no cell asserts the reject (today it's a codegen crash). Add a `check_expr` unit test (each aggregate kind → expectError on `==`) and a conformance `.error` cell. NOTE the open struct-semantics question must be settled with the user before pinning the struct arm's behavior (reject vs fieldwise).

### Defect 6: nested-array `[N][M]T` composite literal / whole-inner-array store mis-stores the inner alloca pointer

**Symptom**: a nested-array literal element (`[2][3]int{ {1,2,3}, {4,5,6} }`) or a whole-inner-array assignment (`a[i] = [M]T{...}` where `a` is `[N][M]T`) stores the inner array's alloca POINTER (8 bytes) into the array-sized inner slot instead of the loaded array VALUE → garbage. (The element-level write `a[i][j]=v` already works — covered by 637.)

**Root cause (confirmed)**: `pkg/binate/ir/gen_composite.bn` `genArrayLit:148-158`. The nested-composite guard loads the value only when `val.Op == OP_ALLOC && val.TypeArg.Kind == types.TYP_STRUCT` (`:154`) AND `elemTyp.Kind == types.TYP_STRUCT` (`:155`). An inner ARRAY literal returns an `arrPtr` whose `TypeArg.Kind == TYP_ARRAY` (the recursive `genArrayLit` allocs an array at `:121`), so the `TYP_STRUCT` guard is false and the alloca pointer is stored raw via `emitStoreManagedSlot`/`EmitStore` (`:164`). The mirror site `gen_control.bn:20-25` `emitArrayElemStore` (shared by the array-index-assign arm at `:253`) has the identical `TYP_STRUCT`-only guard, so `a[i] = [M]T{...}` mis-stores the same way. (The struct-element analogue in `genCompositeLit:96-99` is correct only because struct elements ARE `TYP_STRUCT`.)

**Fix shape**: PATH PARITY — add a `TYP_ARRAY` arm so an inner array literal is value-loaded before the store, exactly as the struct arm loads a struct value. In `genArrayLit:154-158`, broaden the guard: when `val` is an `OP_ALLOC` of `TYP_ARRAY` and `elemTyp.Kind == TYP_ARRAY`, `val = b.EmitLoad(val, val.TypeArg)` (matching the struct load). Mirror in `gen_control.bn:20-25` (`emitArrayElemStore`): add the `TYP_ARRAY` case to the alloca-load guard so the value is loaded when `elemTyp.Kind == TYP_ARRAY`. Keep the pointer-slot case (don't load when the slot holds a pointer-to-array) — only load when the element type is the array value itself, as the struct arm already gates on `elemTyp.Kind == TYP_STRUCT`. Consider a small shared `isAggregateAllocToLoad(val, elemTyp)` predicate covering both `TYP_STRUCT` and `TYP_ARRAY` to keep the two sites in lockstep.

**Files/functions**: `pkg/binate/ir/gen_composite.bn` (`genArrayLit` nested-composite guard ~:154-158); `pkg/binate/ir/gen_control.bn` (`emitArrayElemStore` alloca-load guard ~:20-25).

**Test coverage**: GAP — confirmed no conformance test uses a nested-array LITERAL or whole-inner-array assign (637 covers only element-level `a[i][j]=v`). Add a conformance cell: a `[N][M]T{...}` literal read back + an `a[i] = [M]T{...}` whole-inner-array assign, across all modes (incl. an `@[]T`/managed inner-element variant for refcount).

### Defect 7: compound shift-assign `<<=`/`>>=` bypasses the overshift guard (CONFIRMED: `uint32 y <<= 40` → 256 on LLVM; spec 0)

**Symptom**: `var y uint32 = 1; y <<= 40` yields `256` (40 & 31 = 8 → `1<<8`) on the LLVM backend, not the spec'd `0`. The expression form `y = y << 40` is correct (yields 0); only the compound `<<=`/`>>=` is wrong.

**Root cause (confirmed)**: `pkg/binate/ir/gen_control.bn:51-56` `emitCompoundBinop` computes `op := compoundToIROp(stmt.Op)` (which maps `SHL_ASSIGN`→`OP_SHL`, `SHR_ASSIGN`→`OP_SHR` per `gen_util.bn:46-47`), runs `emitDivCheckGuard` (only for `/=`/`%=`), then `b.EmitBinop(op, cur, rhs, typ)` — a RAW shift with no overshift handling. The expression form routes shifts through the guard: `gen_binary.bn:110-114` calls `emitGuardedShift` for `OP_SHL`/`OP_SHR` whenever the count isn't a safe in-range constant (`isSafeConstShiftCount`, `:131-136`). `emitGuardedShift` (`:148-179`) is the branchless spec-correct lowering (mask-to-zero on overshift; sign-fill clamp for arithmetic `>>`). The compound path never reaches it, so every backend hardware-masks the count.

**Fix shape**: PATH PARITY — route compound shifts through the same guard the expression form uses. In `emitCompoundBinop`, when `op == OP_SHL || op == OP_SHR` and `typ.IsInteger()` and NOT `isSafeConstShiftCount(rhs, typeWidth(typ))`, `return emitGuardedShift(b, op, cur, rhs, typ)` instead of the bare `EmitBinop`; otherwise fall through to the existing `EmitBinop` (the in-range-const fast path). This mirrors `gen_binary.bn:110-114` exactly. `emitGuardedShift`, `isSafeConstShiftCount`, and `typeWidth` are all free functions in the same package (`gen_binary.bn`), directly reachable. The `emitDivCheckGuard` call stays for the `/=`/`%=` case (the shift early-return doesn't reach it, which is correct — a shift never divide-faults).

**Files/functions**: `pkg/binate/ir/gen_control.bn` (`emitCompoundBinop` ~:51-56). Reuse `emitGuardedShift` / `isSafeConstShiftCount` / `typeWidth` from `gen_binary.bn`.

**Test coverage**: GAP — 640 covers `+= -= *= /=` across lvalue forms but NOT `<<=`/`>>=`; 631 covers only expression-form shifts. Add a conformance cell (or extend 640) exercising `<<=`/`>>=` overshift across lvalue forms (variable, array elem, slice elem, field, deref) and both signedness/widths, asserting spec values (0 for logical overshift, sign-fill for arithmetic `>>`).

### Defect 8: transitive re-exported interface resolves to `i8*` (LOWER PRIORITY — reproducer UNCONFIRMED)

**Symptom (as reported)**: an interface re-exported through an intermediate package (consumer imports A, A re-exports interface `I` from B) resolves the bare name `I` to a degraded managed pointer (`i8*`) instead of a 2-word interface value, causing an ABI mismatch at the dispatch site.

**Root cause (suspected, NOT fully confirmed)**: `pkg/binate/ir`. The bare-name interface resolver honors a SINGLE import-alias hop: `isInterfaceTypeExpr` (`gen_iface.bn:244-247`) and `ifaceTypeForName` (`gen_iface.bn:315-321`) consult exactly one `currentImportAlias` (`gen_util.bn`'s `resolveTypeExpr` TEXPR_NAMED arm at `:282-293` likewise tries one alias prefix). `currentImportAlias` is a single string set per-import at `gen_import.bn:158`, `gen_register_import.bn:22`, `gen_module.bn:171` — one level. A transitively-re-exported interface, whose canonical package is two hops away, would miss the single-hop lookup and fall through to `MakeManagedPtrType` (the `@I → @ptr` degradation the in-package self-reference comment at `gen_iface.bn:241-243` warns about). **I could NOT confirm a live reproducer**: there is no `claude-todo.md` entry for transitive-interface re-export, and no conformance test exercises it. ### Defect 9: unary minus `-x` on a sub-word int is mis-typed → invalid IR / silent wrong value (CONFIRMED 2026-06-08; the exact analog of the fixed `~` `bitnot-result-type` bug)

**Symptom**: `-x` for any sub-word int (`uint/int 8/16/32`). Facet A (invalid IR): emits `sub i64 0, %x` with a hardcoded i64 zero while `%x` is i8/16/32 → clang rejects it (`'%x' defined with type 'i8' but expected 'i64'`); unary minus does not compile for sub-word ints on LLVM. Facet B (silent wrong value): on the VM + native backends it computes a host-width negation, so `-1` as `uint8` reads as host `-1`, not `255` (surfaces when the result is consumed without a re-narrow — a `>> k`, an unsigned compare, a widening read). `int64` and float negation are correct (handled); only sub-word is wrong.

**Root cause (confirmed)**: `pkg/binate/ir/gen_expr.bn:223-241` (`genUnary`, MINUS arm) sets `negTyp` defaulting to host-word `types.TypInt()` and overrides it only for floats or `arg.Typ.Width == 64` (the int64-preserving path). A sub-word operand matches neither, so `EmitUnary(OP_NEG, arg, negTyp)` carries i64 while `arg` is i8/16/32 — the SAME mistake `~` had. The fixed `~` entry (`bitnot-result-type`, binate `42ad4fa0`) typed `OP_BITNOT` as the operand's type "mirroring OP_NEG", not realizing OP_NEG had the identical latent gap for sub-word.

**Fix shape**: type the `OP_NEG` result as the operand's resolved (sub-word) type — accept any concrete `TYP_INT` width from the checker resolution (`e.ResolvedTypeID`) / operand type, not just `Width == 64`, mirroring the `~` fix at `gen_expr.bn:247`. A one-site result-type change. The native/VM `OP_NEG` sub-word re-narrow this also needs is ALREADY landed (binate `68616b20`, plan-cr2-3 Defect 1), so once IR-gen types OP_NEG correctly facet B is correct on every backend and facet A compiles.

**Files/functions**: `pkg/binate/ir/gen_expr.bn` (`genUnary` MINUS arm ~:223-241).

**Test coverage**: a `conformance/regressions/unary-minus-subword` cell (`((-x) >> k) == spec` self-check, xfailed every mode until the fix — the `>> k` consume avoids the re-narrow that would mask facet B on a bare compare/store). When this lands, re-add the full `scalar-diff/neg/{8,16,32,64}/{signed,unsigned}` generator family (drafted + reverted out of the plan-cr2-3 Defect-1 commit) — it goes green across all backends once OP_NEG is typed correctly and Defect 1's narrow applies. (Defect 1's OP_NEG narrow is independently pinned by the new aarch64/x64 `emitUnop` narrow unit tests, which build a correctly-typed sub-word OP_NEG directly.) Discovered 2026-06-08 via the plan-cr2-3 Defect-1 neg-cell probe; filed in `claude-todo.md` (MAJOR).
