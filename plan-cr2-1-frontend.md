# Code-Red P2 (CR-2) â€” Plan 1: Front-end IR-gen + type-checker (wrapper-transparency + path-parity defects)

> One of three disjoint Code-Red-2 fix plans. This plan owns `pkg/binate/ir` and `pkg/binate/types` ONLY; it touches no `pkg/binate/codegen`, `pkg/binate/native`, or `pkg/binate/vm` file.
> Every defect below was source-confirmed against the live tree under `/Users/vtl/binate/temp-binate-4/pkg` (line numbers cited are what I found; the earlier todo/plan citations had drifted). Defect-of-record entries live in `explorations/claude-todo.md`.

## Theme

Eight front-end defects, all instances of the two CR-2 meta-invariants. (a) WRAPPER TRANSPARENCY â€” a `Kind`-dispatch that forgets to peel `TYP_READONLY` / `TYP_NAMED` (and `TYP_ALIAS`) before testing the wrapped kind, so a `readonly @Box` / named-distinct value sails past every arm and silently materializes literal-0 (defect 1) or hard-rejects dispatch (defect 2). (b) PATH PARITY â€” a feature lowered in N parallel paths fixed in only some of them: interface-method RESULTS never got the params-flat generalization the PARAMS side did (defect 3); the multi-return DESTRUCTURE checks accept `TYP_FUNC` but not the `TYP_FUNC_VALUE` siblings (defect 4); the nested-composite alloca-load guard fires for struct elements but not array elements (defect 6); the expression-form shift routes through the overshift guard but the compound-assign form does not (defect 7). Defect 5 is a missing checker guard (aggregate `==`/`!=` accepted â†’ invalid `icmp`). Defect 8 (transitive re-exported interface) is the one speculative item â€” flagged below as unconfirmed-reproducer. The unifying repair is the same in every case: route the dropped shape through the machinery the working sibling already uses, after peeling wrappers at the choke point.

## Defects

### Defect 1: `readonly` (and named-wrapped) managed/raw struct-pointer field read silently returns literal-0

**Symptom**: `var p @Box = make(Box); p.v = 7; var rp readonly @Box = p; println(rp.v)` prints `0`, not `7` (direct `p.v` prints `7`). Silent data corruption. **Root cause (confirmed)**: `pkg/binate/ir/gen_selector.bn`. `genSelector` (the field-read path) and `genSelectorPtr` (the lvalue path) dispatch on the *raw* variable type via three Kind-tests that do NOT peel wrappers:

- `gen_selector.bn:30,46` â€” `isManagedPtrToStruct(varTyp)` / `isRawPtrToStruct(varTyp)` (pointer arms), and the parallel sites at `:75,88,106,118,149,162,192,227,238` plus the lvalue mirrors in `genSelectorPtr` at `:313,322,334,352,361,373,389,399,416,425,437`.

The predicates themselves are non-transparent: `gen_refcount_pred.bn:119-123` `isManagedPtrToStruct` tests `t.Kind == types.TYP_MANAGED_PTR && t.Elem.Kind == types.TYP_STRUCT` with no peel; `:126-130` `isRawPtrToStruct` likewise; `:15-18` `isManagedPtrType` likewise. For a `readonly @Box`, `varTyp.Kind == TYP_READONLY`, so all three fail, every arm of `genSelector` is skipped, and control reaches the fallback `gen_selector.bn:297` `return b.EmitConstInt(0, types.TypInt())` â€” literal 0, type-independent. (Verified the parallel `llvmType` choke point peels both `TYP_READONLY` and `TYP_NAMED`/`Underlying` at `pkg/binate/codegen/emit_types.bn:59,61-62`, so codegen would have emitted the right LLVM type had IR-gen produced the GEP â€” the bug is purely that IR-gen never produces it.)

**Fix shape**: Make the predicates wrapper-transparent at the choke point â€” peel `TYP_READONLY` and `TYP_NAMED` (mirroring `llvmType`) inside `isManagedPtrToStruct`, `isRawPtrToStruct`, and `isManagedPtrType`. ### Defect 5: `==` / `!=` on an aggregate operand is accepted by the checker, then emits invalid `icmp`

**Symptom**: `a == b` / `a != b` where operands are `*[]int`, `@[]int`, `*func()int`, `@func()int`, `@Iface`/`@errors.Error`, or a struct `Pt` type-checks, then codegen emits a scalar `icmp` on a >1-word value â†’ clang `error: icmp requires integer operands`; whole package fails. `claude-todo.md` CONFIRMED; the *semantics* are an open spec gap, the user's direction (2026-06-07) is DISALLOW for interface values, slices, and func values.

**Root cause (confirmed)**: `pkg/binate/types/check_expr.bn:262-269`. `checkBinaryOp`'s comparison branch (covers `==`/`!=`/`<`/`>`/`<=`/`>=`) does only `if !lt.AssignableTo(rt) && !rt.AssignableTo(lt) { errTypeMismatch }` then `return TypUntypedBool()` â€” there is NO scalar/pointer operand guard. So two assignable aggregate operands pass, and codegen's equality lowering (which assumes a single-word integer/pointer and emits `icmp`) produces invalid IR.

**Fix shape**: Add a checker reject for `==`/`!=` on aggregate operands (the user-chosen disposition; this plan implements the agreed DISALLOW, it does not invent new equality semantics). In the `==`/`!=` sub-branch of `check_expr.bn:262-269`, after the type-compatibility check, reject when either operand is non-scalar-and-non-pointer: i.e. NOT (`IsScalar()` or `IsPointer()` or `TYP_NIL`). `IsScalar()` (`types_query.bn:207-209`, covers int/float/bool through named/readonly) and `IsPointer()` (`:212-218`, raw + managed pointer) already exist; `nil` comparisons against a pointer stay allowed. This rejects slices, managed-slices, func-values, interface-values, and structs with a clear diagnostic (e.g. "cannot compare values of aggregate type T; use present() for interface/slice emptiness"). Scope per the user decision: DISALLOW iface/slice/func-value definitively; STRUCT equality is still an open spec question (fieldwise vs disallow) â€” flag to the user whether to reject structs now or defer the struct arm. Either way the current invalid-IR path must end (reject or correct-lowering, never `icmp` on an aggregate). The ordering relational ops (`<`/`>`/`<=`/`>=`) on aggregates should also be rejected here (they're already nonsensical); confirm no test relies on them.

**Files/functions**: `pkg/binate/types/check_expr.bn` (`checkBinaryOp` comparison arm ~:262-269). Reuse `IsScalar`/`IsPointer` from `types_query.bn`.

**Test coverage**: GAP â€” no cell asserts the reject (today it's a codegen crash). Add a `check_expr` unit test (each aggregate kind â†’ expectError on `==`) and a conformance `.error` cell. NOTE the open struct-semantics question must be settled with the user before pinning the struct arm's behavior (reject vs fieldwise).

### Defect 6: nested-array `[N][M]T` composite literal / whole-inner-array store mis-stores the inner alloca pointer

**Symptom**: a nested-array literal element (`[2][3]int{ {1,2,3}, {4,5,6} }`) or a whole-inner-array assignment (`a[i] = [M]T{...}` where `a` is `[N][M]T`) stores the inner array's alloca POINTER (8 bytes) into the array-sized inner slot instead of the loaded array VALUE â†’ garbage. (The element-level write `a[i][j]=v` already works â€” covered by 637.)

**Root cause (confirmed)**: `pkg/binate/ir/gen_composite.bn` `genArrayLit:148-158`. The nested-composite guard loads the value only when `val.Op == OP_ALLOC && val.TypeArg.Kind == types.TYP_STRUCT` (`:154`) AND `elemTyp.Kind == types.TYP_STRUCT` (`:155`). An inner ARRAY literal returns an `arrPtr` whose `TypeArg.Kind == TYP_ARRAY` (the recursive `genArrayLit` allocs an array at `:121`), so the `TYP_STRUCT` guard is false and the alloca pointer is stored raw via `emitStoreManagedSlot`/`EmitStore` (`:164`). The mirror site `gen_control.bn:20-25` `emitArrayElemStore` (shared by the array-index-assign arm at `:253`) has the identical `TYP_STRUCT`-only guard, so `a[i] = [M]T{...}` mis-stores the same way. (The struct-element analogue in `genCompositeLit:96-99` is correct only because struct elements ARE `TYP_STRUCT`.)

**Fix shape**: PATH PARITY â€” add a `TYP_ARRAY` arm so an inner array literal is value-loaded before the store, exactly as the struct arm loads a struct value. In `genArrayLit:154-158`, broaden the guard: when `val` is an `OP_ALLOC` of `TYP_ARRAY` and `elemTyp.Kind == TYP_ARRAY`, `val = b.EmitLoad(val, val.TypeArg)` (matching the struct load). Keep the pointer-slot case (don't load when the slot holds a pointer-to-array) â€” only load when the element type is the array value itself, as the struct arm already gates on `elemTyp.Kind == TYP_STRUCT`. Consider a small shared `isAggregateAllocToLoad(val, elemTyp)` predicate covering both `TYP_STRUCT` and `TYP_ARRAY` to keep the two sites in lockstep.

**Files/functions**: `pkg/binate/ir/gen_composite.bn` (`genArrayLit` nested-composite guard ~:154-158); `pkg/binate/ir/gen_control.bn` (`emitArrayElemStore` alloca-load guard ~:20-25).

### Defect 7: compound shift-assign `<<=`/`>>=` bypasses the overshift guard (CONFIRMED: `uint32 y <<= 40` â†’ 256 on LLVM; spec 0)

**Symptom**: `var y uint32 = 1; y <<= 40` yields `256` (40 & 31 = 8 â†’ `1<<8`) on the LLVM backend, not the spec'd `0`. The expression form `y = y << 40` is correct (yields 0); only the compound `<<=`/`>>=` is wrong.

**Root cause (confirmed)**: `pkg/binate/ir/gen_control.bn:51-56` `emitCompoundBinop` computes `op := compoundToIROp(stmt.Op)` (which maps `SHL_ASSIGN`â†’`OP_SHL`, `SHR_ASSIGN`â†’`OP_SHR` per `gen_util.bn:46-47`), runs `emitDivCheckGuard` (only for `/=`/`%=`), then `b.EmitBinop(op, cur, rhs, typ)` â€” a RAW shift with no overshift handling. The expression form routes shifts through the guard: `gen_binary.bn:110-114` calls `emitGuardedShift` for `OP_SHL`/`OP_SHR` whenever the count isn't a safe in-range constant (`isSafeConstShiftCount`, `:131-136`). `emitGuardedShift` (`:148-179`) is the branchless spec-correct lowering (mask-to-zero on overshift; sign-fill clamp for arithmetic `>>`). The compound path never reaches it, so every backend hardware-masks the count.

**Fix shape**: PATH PARITY â€” route compound shifts through the same guard the expression form uses. In `emitCompoundBinop`, when `op == OP_SHL || op == OP_SHR` and `typ.IsInteger()` and NOT `isSafeConstShiftCount(rhs, typeWidth(typ))`, `return emitGuardedShift(b, op, cur, rhs, typ)` instead of the bare `EmitBinop`; otherwise fall through to the existing `EmitBinop` (the in-range-const fast path). This mirrors `gen_binary.bn:110-114` exactly. `emitGuardedShift`, `isSafeConstShiftCount`, and `typeWidth` are all free functions in the same package (`gen_binary.bn`), directly reachable. The `emitDivCheckGuard` call stays for the `/=`/`%=` case (the shift early-return doesn't reach it, which is correct â€” a shift never divide-faults).

**Files/functions**: `pkg/binate/ir/gen_control.bn` (`emitCompoundBinop` ~:51-56). Reuse `emitGuardedShift` / `isSafeConstShiftCount` / `typeWidth` from `gen_binary.bn`.

**Test coverage**: GAP â€” 640 covers `+= -= *= /=` across lvalue forms but NOT `<<=`/`>>=`; 631 covers only expression-form shifts. Add a conformance cell (or extend 640) exercising `<<=`/`>>=` overshift across lvalue forms (variable, array elem, slice elem, field, deref) and both signedness/widths, asserting spec values (0 for logical overshift, sign-fill for arithmetic `>>`).
